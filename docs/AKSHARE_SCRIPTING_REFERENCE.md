## 为什么单独记录这份文档

`AKShare` 官网首页更适合作为**总索引**，而不是直接当项目说明书来用。  
对 `etf-report` 来说，这份文档的目标是把“去哪里查、查到后怎么接进脚本、接入时要注意什么”沉淀下来，减少每次写新数据脚本都重新摸索。

当前项目里，`AKShare` 已经是正式数据源之一：
- `scripts/corporate_action_source.py`：使用 `akshare.fund_cf_em()` 拉取基金拆分 / 份额折算事件
- `DESIGN.md`：已把 `AKShare fund_cf_em` 记入主流程的数据获取层

## 把首页当成什么来用

把 `https://akshare.akfamily.xyz/index.html` 当成 **AKShare 文档总目录**：
- **先按数据主题找入口**：股票、基金、指数、宏观、期货、债券、港股、美股等
- **再进入具体接口页**：看参数、返回列名、示例代码、字段语义
- **最后再回到项目里做标准化**：不要把原始列名直接散落到业务逻辑中

对我们来说，首页最重要的价值不是“复制全文”，而是提供：
- **接口发现入口**
- **参数与字段说明入口**
- **示例代码入口**
- **按主题扩展新数据源的路线图**

## `etf-report` 写脚本时的推荐查找路径

### 场景 1：已经知道业务主题，不知道接口名

推荐顺序：
1. 打开首页：`https://akshare.akfamily.xyz/index.html`
2. 进入最接近的主题栏目（基金 / 指数 / 宏观 / 股票等）
3. 在栏目页里找接口说明、输入参数、输出字段、示例
4. 确认返回字段是否足够支持项目需求
5. 再决定是否接入 `etf-report`

### 场景 2：已经知道疑似接口名

推荐顺序：
1. 先在官网文档中定位接口页
2. 重点看 4 件事：
   - 输入参数是否可控
   - 输出列名是否稳定
   - 日期字段格式是否清楚
   - 是否支持一次拉全量 / 按时间窗口过滤
3. 通过后再写脚本和测试

### 场景 3：要给现有链路补充结构化事件

优先考虑 `AKShare`，因为它更适合：
- 提供**结构化表格数据**
- 明确列出字段语义
- 作为“辅助事件流”并入主流程

如果是下面这些场景，`AKShare` 往往比直接抓网页更合适：
- 基金拆分 / 份额折算
- 指数或基金补充元数据
- 可结构化的统计类 / 排名类数据
- 适合按 DataFrame 处理的中间层数据

## 当前已落地接口：`fund_cf_em`

### 适用场景

用于识别基金拆分 / 份额折算等会影响历史价格口径的事件，供数据清洗主流程消费。

### 当前项目中的接入方式

`scripts/corporate_action_source.py` 当前使用：
- 接口：`akshare.fund_cf_em()`
- 关键输入：`year`、`typ`、`rank`、`sort`、`page`
- 关键输出列：
  - `基金代码`
  - `基金简称`
  - `拆分折算日`
  - `拆分类型`
  - `拆分折算`

### 在项目里的标准化结果

项目不会把原始字段直接传到业务层，而是统一转换成：
- `action`
- `ex_date`
- `ratio`
- `raw_type`
- `source`
- `note`

这样做的好处是：
- 上游字段名改动更容易集中修复
- 主流程只消费标准结构，不依赖中文列名
- 测试更容易写，也更容易 mock

## 推荐脚本模式

### 1. 先调用接口，再立即判空

```python
import akshare as ak

df = ak.fund_cf_em(year="2026")
if df is None or df.empty:
    return []
```

### 2. 把文档列名尽快映射成项目内部字段

```python
row = {
    "code": str(raw["基金代码"]).zfill(6),
    "ex_date": str(raw["拆分折算日"])[:10],
    "ratio": float(raw["拆分折算"]),
}
```

### 3. 在进入主流程前做过滤和标准化

建议至少处理：
- ETF 白名单过滤
- 日期窗口过滤
- 类型映射
- 数值合法性过滤
- 去重

### 4. 保存结构化产物，便于审计和回放

当前项目已经采用这种方式：
- 输出文件：`data/corporate_action_events.json`
- 目的：便于审计、调试、回放与回退

## 接入 `AKShare` 新接口时的检查清单

### 接口选择前
- 这个接口是否来自官网文档而不是二手文章
- 参数和字段说明是否完整
- 是否能稳定表达我们需要的业务语义
- 是否需要分页、按年抓取、或二次筛选

### 编码时
- 不要把中文原始列名散落到多个脚本里
- 统一在一个 source 模块里做原始字段映射
- 记录 `source` 字段，便于审计
- 遇到异常时要能优雅降级，不能把主流程直接拖死

### 验证时
- 覆盖空结果
- 覆盖异常字段 / 非法日期 / 非法数值
- 覆盖窗口过滤
- 覆盖重复记录去重
- 覆盖主流程消费结果是否正确

## `etf-report` 中的使用边界建议

### 适合优先考虑 `AKShare` 的场景
- 结构化基金事件
- 指数 / 基金补充元数据
- 排名、统计、清单型数据
- 适合单独生成 JSON 供主流程消费的数据

### 不建议轻易替换为 `AKShare` 的场景
- 已经在主流程中稳定运行的日 K / 周 K 主行情链
- 已有稳定来源且替换成本高的实时接口
- 对时效和一致性高度敏感、但当前链路已经稳定的关键主数据

原则上：
- **AKShare 更适合作为结构化辅助源**
- **主行情源替换必须单独评估，不顺手就改**

---

## 常见坑位

- **接口文档是入口，不是最终契约**：上线前一定要跑真实数据验证
- **列名多为中文**：务必在 source 层统一映射
- **日期字段要先标准化**：统一到 `YYYY-MM-DD`
- **数值字段要显式转型**：尤其是比例、金额、成交量
- **空表要有兜底**：不能把空 DataFrame 直接带进主流程
- **保留原始语义**：如 `raw_type` 这类字段在排查时很有价值

## 给未来写脚本时的最短行动建议

遇到新的外部数据需求时，按下面顺序做：
1. 先去 `AKShare` 首页找主题入口
2. 找到具体接口页，看参数 / 字段 / 示例
3. 判断它是主数据源还是辅助结构化源
4. 优先写成独立 source 模块
5. 输出标准 JSON，再让主流程消费
6. 补测试，不让业务逻辑直接依赖原始列名
