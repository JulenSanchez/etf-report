# AKShare 候选接口清单

**创建日期**: 2026-04-16  
**适用范围**: `etf-report` 后续外部数据源扩展评估  
**配套文档**: `docs/AKSHARE_SCRIPTING_REFERENCE.md`

---

## 这份清单怎么用

这不是“看到接口就马上接”的待办表，而是给 `etf-report` 写脚本时的**候选雷达**。

阅读方式：
- **高置信候选**：已经能对应到明确的官方接口名或官方栏目页，可直接进入技术评估
- **栏目级候选**：已经确认官方文档有对应栏目，但具体接口名需要在真正接入前再二次核验

核心原则：
- **优先补结构化辅助源**，不要轻易动已经稳定运行的主行情链
- **先做 source 模块，再让主流程消费**
- **接入前必须看官方文档 + 跑真实数据 + 补测试**

## 一、从项目现状反推的候选数据空位

| 空位 | 当前状态 | 价值 | 优先级建议 |
|------|----------|------|------------|

## 二、高置信候选（可直接进入技术评估）

### 1. `fund_cf_em` —— 基金拆分 / 份额折算

- **当前状态**: 已落地
- **官方文档页**: `https://akshare.akfamily.xyz/data/fund/fund_public.html`
- **适合场景**:
  - ETF 份额分拆
  - 份额折算
  - 历史价格清洗前的结构化事件识别
- **已在项目中的定位**:
  - `scripts/corporate_action_source.py`
  - 输出 `data/corporate_action_events.json`
- **结论**:
  - 这是当前 `etf-report` 与 `AKShare` 最成功的一条接入范式，后续新接口优先复用这套“source 模块 + 标准 JSON + 主流程消费”的设计。

### 2. `fund_name_em` —— 基金基础信息总表入口

- **官方文档页**: `https://akshare.akfamily.xyz/data/fund/fund_public.html`
- **官方文档搜索锚点**: 文档搜索结果已明确出现 `接口: fund_name_em`
- **适合场景**:
  - ETF / 基金代码与名称校验
  - 新 ETF 加入池子时做基础信息核对
  - 生成更稳定的基金别名 / 展示名映射
- **适合本项目的位置**:
  - 作为 `config/config.yaml` 的辅助校验源
  - 可单独生成一份基金基础信息缓存 JSON，不直接塞进主流程
- **接入建议**:
  - 只把它当“元数据校验 / 补全源”，不要替代当前人工确认过的 ETF 池配置

### 3. `stock_zh_index_spot_em` —— 指数实时行情（东财）

- **官方文档页**: `https://akshare.akfamily.xyz/data/index/index.html`
- **官方文档搜索锚点**: 文档搜索结果已明确出现 `接口: stock_zh_index_spot_em`
- **适合场景**:
  - 沪深 300 等基准指数的实时 / 当日快照校验
  - 新浪指数链路异常时的备用参考
  - 主行情链与备用源之间做交叉检查
- **适合本项目的位置**:
  - 先作为“备用源 / 校验源”而不是主替代
  - 可用于健康检查或调试脚本，不建议直接替换当前稳定链路
- **接入建议**:
  - 先做只读比对脚本，确认字段与时间口径稳定后，再讨论是否进入正式流程

## 三、栏目级候选（方向明确，接入前二次核验具体接口）

### 1. 公募基金数据栏目

- **官方栏目页**: `https://akshare.akfamily.xyz/data/fund/fund_public.html`
- **适合扩展的方向**:
  - 基金基础档案
  - ETF / 公募基金行情补充
  - 排名 / 统计 / 事件类结构化数据
- **为什么值得关注**:
  - 当前项目已经证明该栏目里的 `fund_cf_em` 可用
  - 后续若要补基金元数据或辅助解释层，优先从这里继续找
- **建议动作**:
  - 真正开发前，在该栏目页内继续核对是否存在适合的 ETF 行情、基金档案、资金相关接口

### 2. 指数数据栏目

- **官方栏目页**: `https://akshare.akfamily.xyz/data/index/index.html`
- **适合扩展的方向**:
  - 基准指数实时快照
  - 指数日线 / 历史行情备用源
  - 指数统计指标
- **为什么值得关注**:
  - `etf-report` 当前所有 ETF 都依赖指数对标逻辑
  - 即便不替换主链，也很适合做备用校验层
- **建议动作**:
  - 后续若遇到新浪指数链路不稳，优先从这里找日频 / 实时指数接口做备胎

### 3. 宏观数据栏目

- **官方栏目页**: `https://akshare.akfamily.xyz/data/macro/macro.html`
- **搜索结果确认**: 官方页面标题已明确存在 `AKShare 宏观数据`
- **适合扩展的方向**:
  - CPI / PPI / PMI / GDP 等宏观背景数据
  - 日报 / 周报中“市场环境解释”模块
  - 把纯主观宏观点评，升级为“有数据锚点的解释”
- **适合本项目的位置**:
  - 更适合作为**解释层补充**，不适合插进主行情主链
- **接入建议**:
  - 宏观接口较多，务必按实际文案需要反推指标，不要为了“有宏观数据”而盲接

### 4. 北向资金 / 沪深港通相关栏目

- **官方文档线索**: 当前公开搜索能确认 `AKShare` 文档体系和历史接口中存在北向资金相关内容，但本轮没有稳定拿到新版官网的精确页面锚点
- **适合扩展的方向**:
  - 北向资金净流入
  - 沪深港通持股变化
  - 市场情绪辅助解释
- **为什么值得关注**:
  - `etf-report` 页面里已有“北向资金图”语义，后续如果要让图表数据更结构化，这是高价值方向
- **接入建议**:
  - 暂列为“值得追踪”的候选项；真正开发前，再去官网栏目页核一次接口名和返回字段

## 四、建议的接入顺序

### 第一梯队：最值得优先做

1. **基金流向 / 资金流向**
   - 原因：`fund_flow_data.json` 已预留，项目天然有落点
   - 目标：补出可独立生成的数据文件，供页面解释层或图表层消费

2. **指数备用源 / 校验源**
   - 原因：风险收益比高，不影响主链，先做辅助校验最稳
   - 目标：给基准指数链路增加第二事实源

### 第二梯队：适合增强说明层

3. **基金基础信息 / 档案元数据**
   - 原因：实现成本相对可控，能减少静态配置维护压力
   - 目标：自动校验名称、代码、可能的补充说明字段

4. **宏观补充数据**
   - 原因：更适合做报告解释层升级，而不是底层抓数改造
   - 目标：给宏观分析段落增加客观数据锚点

### 第三梯队：谨慎推进

5. **成分股自动补充 / 自动维护持仓**
   - 原因：收益高，但口径、时效、稳定性要求也高
   - 目标：先做辅助校验，不要直接替代 `holdings.yaml`

## 五、每个候选接口进入开发前都要过的门槛

- **门槛 1**: 能在官方文档里找到明确说明
- **门槛 2**: 参数和返回字段足够稳定
- **门槛 3**: 能独立写成 source 模块
- **门槛 4**: 能输出标准 JSON，而不是把原始 DataFrame 直接传下去
- **门槛 5**: 有空数据兜底、异常降级和最小测试覆盖

如果做不到这 5 条，就先不要进主流程。

---

## 六、对 `etf-report` 的现实建议

如果下一步真的要做，我建议优先顺序是：

1. **先补基金流向候选调研**：把 `fund_flow_data.json` 变成真正产物  
2. **再补指数备用源比对脚本**：不替换主链，只做交叉校验  
3. **最后再考虑宏观和基金档案补充**：放到解释层，不打断当前日报稳定性

---

**结论**: 目前最适合 `etf-report` 的 `AKShare` 扩展方向，不是“全面替换现有数据源”，而是继续沿着 **结构化辅助数据** 往前走：优先补 `fund_flow`、指数备用校验、基金元数据和宏观解释层。
