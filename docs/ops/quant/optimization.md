# 参数优化全流程约定

> **触发词**: 用户说"优化 <preset>"。AI 自动：自检数据 → 推导搜索空间 → 执行 → 出 analysis.json → 写报告。详见本文。

AI 与用户的协作契约。优化启动前双方各司其职，减少反复沟通。

## 一、用户如何发起

### 最小提示词

```
优化 <preset>
```

例：`优化 gam-2`、`优化 zen-1`

AI 应自动从 `PRESET_OPT_PROFILES` 读取 metric 和约束，从 preset 当前值推导搜索范围，无需用户指定。

### 完整提示词（覆盖默认值）

```
优化 <preset>，目标 <metric>，约束 <constraint>，<n> trials，<周期>
```

例：`优化 gam-2，目标 total_return，约束 mdd,-15，100 trials，6Y`

### 多 preset 批量

```
优化 gam-2 zen-1 act-1，70 trials，6Y
```

AI 自动串行执行，每完成一个立即写报告。

## 二、AI 执行前自检（AI 必须逐条确认）

| # | 检查项 | 动作 |
|---|--------|------|
| 1 | 数据是否新鲜？ | `python scripts/quant_data_fetcher.py` 确认 CSV 最新日期为最近交易日 |
| 2 | 盘中还是盘后？ | 盘中（9:30-15:10）→ 警告用户"回测会包含不完整日内数据" |
| 3 | Config 是否有未提交的改动？ | `git diff config/quant_universe.yaml` 检查，有改动 → 询问是否先提交 |
| 4 | 搜索范围覆盖基线值？ | 手动验证 `--params` 中每个参数的 min/max 包含基线值 |
| 5 | 权重参数 sum=100？ | 搜索的 w1/w3 范围 + 固定 w7 必须 sum=100，否则 residual 会调整 |
| 6 | disc_step 精度？ | 如果预设 disc_step 不是整数百分点，确认滑块 max 足够 |

## 三、AI 执行规范

### 默认参数

- `--strategy bayesian`
- `--n-trials 70`
- `--periods 6Y`
- `--universe '*'`
- `--auto-bounds`
- metric 和约束 → 从 `PRESET_OPT_PROFILES` 读取

### 搜索参数（AI 自行决定，告知用户）

AI 从 preset 当前值推导 8 个关键参数的窄范围：
- w1, w3: [baseline-10, baseline+10]
- max_holdings: [3, 7]
- disc_step: [0.03, 0.15]
- vol_window: [baseline-15, baseline+15]
- dead_zone: [15, 35]
- f7_t: [3, 15]（或 [baseline-3, baseline+5] 若 baseline 很大）
- ma_bear_pos: [0.3, 1.0]（赌徒）/ [0.15, 0.5]（精算师）

其余参数固定基线值。**告知用户搜索空间后开始执行，不等确认。**

### 权重处理

w1、w3、w7 中：w1 和 w3 搜索，w7 固定基线值。`auto_bounds` + `--params` 正确写法：
```
w1=48:65:1 w3=18:30:1 ... w7=20 bias=0 conf_type=ma_trend ...
```
固定值参数用单值（如 `w7=20`），搜索参数用范围（如 `w1=48:65:1`）。

### 并行限制

不同 preset 的优化必须**串行**。同一 preset 的 trial 也只能串行（GIL 限制）。

## 四、AI 完成后交付

### 必须交付物

1. `research/params/<preset>-<date>/analysis.json`（运行 `optimization_analyzer.py` 生成）
2. `research/params/<preset>-<date>/report.md`（基于 analysis.json 写叙事报告）
3. 口头汇报关键结论

### 报告质量标准

- 每节都有"用户读完后能回答什么问题"的检验
- ETF 四象限图必须出现（不是罗列，是分类 + 解读）
- 收敛轨迹要讲故事
- 必须有"是否建议采纳"的明确判断及理由

### 不自动做的事

- **不自动写入 config**（等用户确认）
- **不自动提交 git**（等用户确认）
- **不自动删除旧 preset**

## 五、故障排查

| 症状 | 原因 | 处理 |
|------|------|------|
| 优化器 TR 远低于基线 | 权重 sum≠100，或搜索范围不覆盖基线 | 检查 resolved 权重 sum，检查 auto_bounds 范围 |
| Tuner 前端 TR 与 CLI 不一致 | 1) Tuner 缓存旧 config 2) disc_step slider 截断 3) 日内数据 | 重启 Tuner，检查 slider max，等收盘 |
| `--constraint` 不生效 | argparse 格式错误 | 确认 `--constraint mdd,-20` 无空格、无 % 号 |
| 优化器 OOM | 54 支全池 + preload 数据 ~2GB | 减少 trial 数或重启进程 |
| resolve_weights 改变所有权重 | 固定值参数被当作 weight 类型 | 确认 w7 用 `w7=20` 格式（非 weight type）|

## 六、环境约定

| 事项 | 约定 |
|------|------|
| 数据刷新 | 优化前必须拉最新 CSV（`quant_data_fetcher.py`） |
| 盘中优化 | 警告但不阻止——用户可能想对比日内 vs 收盘结果 |
| git 状态 | dirty working tree → 必须先提交或 stash，否则不跑 |
| Tuner 端口 | 优化器不绑端口，不受 `_ensure_tuner` 影响 |
| 输出目录 | `research/params/<preset>-YYYYMMDD/`，自动创建 |

## 一、分析哲学

### 发现驱动，而非验证驱动

报告的价值在于**从数据中发现用户没说的洞察**。如果用户已经说了"金融科技被盘活"，你只需要验证——那报告就只是整理。好的报告应该：

- 从 TPE 收敛轨迹、参数分布、回测行为中**主动挖掘**值得讲的发现
- 用户给的观察是**起点的线索**，不是终点的结论
- 每节都要问自己：这节有没有讲出用户自己看不出来的东西？

### 分析师视角

把优化结果当成一次**策略体检**。你看到的不仅是"TR 涨了 300pp"——你要解释**为什么涨了、靠什么涨的、涨在哪个阶段、有什么风险**。

## 二、报告结构（8 节）

每节都标注了"自己能否看出"——如果答案是"能"，说明这节只是数据整理，需要深挖。

### 1. 优化概览

读者能自己看出来吗：**能**

- 方法、trial 数、搜索空间
- 基线 vs Top 5 表格
- 采纳参数的最终选择

**质量标准**：一页说清楚。不要长篇。

### 2. 收敛轨迹与搜索洞察

读者能自己看出来吗：**部分能**

- Best-so-far 在 trial 序列上的演进——参数如何逐步逼近最优
- TPE 是否收敛？是否有"顿悟"跳跃？
- 搜索分为几个阶段？每个阶段 TPE 在探索什么方向？
- 对照参数空间：哪些参数搜索了但没效果？（反直觉的发现）

**质量标准**：这里要讲一个"搜索如何找到答案"的故事。如果只是再列一遍数据，删掉。

### 3. 参数敏感性与交互

读者能自己看出来吗：**不能**（需要控制变量验证）

- Top vs Bottom trial 的参数差异——什么参数真正区分了优劣
- 关键参数的凸性/单调性验证（单变量扫）
- 有没有参数"意想不到地不重要"？（如 ma_bear_pos 在 top/bot 无差异）
- 有没有参数存在交互？（如 w1 和 dead_zone 的联合变化）

**质量标准**：不仅说"w1 重要"，要说"w1 从 50 到 64 的过程里，策略行为发生了什么变化"。

### 4. 市场阶段归因

读者能自己看出来吗：**部分能**

- 新策略在牛/熊/震荡/复苏各阶段的表现
- 改善是均匀的还是集中在某个阶段？
- 如果集中在某个阶段——那个阶段有什么特征？策略利用了什么？

**质量标准**：不仅说"全阶段都赢了"，要说赢在什么地方、为什么那个地方能赢。

### 5. 策略行为诊断

读者能自己看出来吗：**不能**

这是报告最有价值的部分。需要深入到：

- 持仓结构如何变化（集中度分布、扇区权重迁移）
- 交易节奏如何变化（换仓频率、持仓时长、进出时点）
- 新旧策略在不同 market regime 下**行为差异**（非结果差异）

**方法论**：选一组代表性的 trial（最优、中游、最差），对比它们的 ETF 选择、扇区偏好、持仓节奏，找出**行为模式**。

**质量标准**：用户读完后能回答"新策略在操作上和老策略有什么不同"。

### 6. 极端场景分析

读者能自己看出来吗：**部分能**

- 极端集中（持仓=1 或 2）的频率、标的、收益
- 最大回撤期间的持仓：策略在暴跌时拿了什么？躲开了吗？
- 最高收益期间的持仓：策略在暴涨时拿了什么？
- 单 ETF 视角：有没有"救世主"（靠它独撑）或"黑洞"（持续拖累）？

**质量标准**：不仅说"极端持仓占比 4%"，要说这些极端情况的**决策逻辑**是什么、实盘是否可接受。

### 7. ETF 层面的结构性发现

读者能自己看出来吗：**不能**（需要从信号历史中提取）

- 按 P&L / 选中率两个维度做 ETF 四象限分类
- 哪些 ETF 是"高频高收益"（核心持仓）、"高频低收益"（应该剔除）、"低频高收益"（狙击手）、"低频低收益"（边缘）
- 扇区内部的 ETF 竞争：同一扇区内，谁替代了谁？
- 新参数是否改变了 ETF 的"命运"（从负贡献到正贡献，或反之）

**质量标准**：用户读完后能回答"哪些 ETF 在新策略里是核心、哪些应该考虑淘汰"。

### 8. 风险提示与后续方向

读者能自己看出来吗：**不能**

- 本次优化的局限性（搜索范围、固定参数、池子依赖）
- 回测在实盘中最可能失效的场景（black swan、regime change）
- 后续值得做的方向（不是泛泛的"继续优化 preset X"，要具体）

## 三、通用研究方法

这些方法不限于 gam-1，适用于任何参数优化。

### 方法 1：收敛轨迹分析

看 best-so-far 的 trial 序列。TPE 是否在"爬山"？是否有平台期后的跳跃？每次跳跃对应哪些参数变化？讲出"搜索的故事"。

### 方法 2：Top-Bottom 分化

不是比 #1 和 #70——是比 Top 10 和 Bottom 10 的参数均值差异。哪些参数真正区分了优劣，哪些没有。后者往往更有洞察（"原来这个参数不重要"）。

### 方法 3：控制变量验证

单个参数在最优值附近的扫参（固定其他参数）。确认是否真的在凸性顶点。注意：TPE 可能找到了局部最优而非全局最优。

### 方法 4：行为对比

取最优、中游、最差 trial 分别跑完整回测，提取持仓序列、扇区权重时序、交易日志。不看结果（TR），看**行为**（选了什么、什么时候选、持有多久）。

### 方法 5：ETF 四象限

P&L 和选中率两个维度交叉：
- 高 P&L + 高选中 = 核心持仓
- 高 P&L + 低选中 = 狙击手（时机精准但出手少）
- 低 P&L + 高选中 = 需要排查（是不是被错误信号反复引诱）
- 低 P&L + 低选中 = 边缘（策略不用它，没影响）

### 方法 6：极端场景拆解

不要只看均值。看"最惨的时刻"（最大回撤期间持仓）和"最辉煌的时刻"（最快上涨期间持仓）。实盘关心的是极端，不是均值。

## 四、搜索空间设计原则

- 先验证 PARAM_BOUNDS 覆盖基线值（c_sensitivity、ema_period 是历史 bug）
- 已调优的 preset 用窄范围 `[cur×0.5, cur×2.0]`
- 低敏参数（corr<0.15）固定在基线
- 权重参数（w1/w3/w7）sum=100%，最后一个用 residual
- 单周期（6Y）可替代多周期，速度 3 倍且结果一致
- `--universe '*'` 确保全池

### 三派优化目标与约束

`quant_contract.py` 的 `PRESET_OPT_PROFILES` 定义了各 preset 的目标和约束：

| Preset | 信仰 | metric | 约束 | 理由 |
|--------|------|--------|------|------|
| gam-1/2 | 赌徒 | annual_return | mdd,-20 | 追求绝对收益，回撤容忍 20% |
| zen-1 | 禅修者 | sharpe | — | 风险调整收益，无需额外约束 |
| act-1 | 精算师 | calmar | bear,0.15,0.30 | 风险调整后收益，熊仓不超 30% |

优化命令示例：
```bash
# 自动使用 preset 默认 metric + 约束
python scripts/quant_optimizer.py --preset gam-2 --metric annual_return --constraint mdd,-20 ...

# 覆盖默认
python scripts/quant_optimizer.py --preset act-1 --metric sharpe --constraint bear,0.15,0.30 ...
```

约束使用 `--constraint type,val` 语法（可重复）：
- `mdd,-20`：MDD 必须 ≥ -20%（超限 trial 被 prune）
- `bear,0.15,0.30`：ma_bear_pos 必须在 [0.15, 0.30]

## 五、优化产物归档规范

每次正式参数优化应形成一个自包含目录：

```text
research/params/<preset>-YYYYMMDD[-vN]/
```

| 文件 | 是否提交 | 用途 |
|---|---|---|
| `report.md` | 是 | 人类可读的最终分析报告 |
| `results.json` | 是 | 优化器输出的结构化结果 |
| `analysis.json` | 是 | `optimization_analyzer.py` 输出，作为报告叙事依据 |
| `log.txt` | 是 | 运行过程、参数空间、异常和完成记录 |
| `.optimizer_done` | 可选 | 自动化完成标记；如用于恢复/判断完成态则提交 |
| `optuna.db` | 否 | 本地可复现材料，体积和可变性较高，不提交 |

若同一天多轮搜索，使用 `-v2` / `-v3` / `-v4` 表示搜索轮次，不表示生产策略版本。最终采纳报告应在总览文档中说明“哪一轮被采纳、为什么不是机械取 best trial”。

写报告前必须先生成 `analysis.json`：

```bash
python scripts/optimization_analyzer.py --study <name> --preset <preset> --baseline-preset <baseline> --top-n 3 --output research/params/<dir>/analysis.json
```

报告应基于 `analysis.json` 写结论；不要只从 `optuna.db` 或 Top10 表里抄数字。

## 六、常见陷阱

- `auto_bounds` 可能给错范围，先手动验证
- `--params` 不加 `--auto-bounds` 时非列表参数走默认值而非基线
- ThreadPoolExecutor 对 CPU 密集回测无效（GIL）
- 权重参数用固定值时不被识别为 weight，不走 residual 调整
- `report.md` 的 Top 表如出现 `0.0000` 指标，说明旧 report generator 字段不完整，应以 `analysis.json` 和最终人工报告为准
