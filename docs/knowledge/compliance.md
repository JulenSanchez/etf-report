# 外部数据合规入门

> 面向**技术经验少、想做个人投资/量化项目的普通人**。
>
> 目标：**不懂专业法律的情况下，也不会因"顺手爬个数"踩大雷**。

---

## 一、先讲清楚三个最常被混淆的概念

### 1. 技术可行 ≠ 合法

很多东西你**能爬**不代表**能用**。Python 几行代码就能抓下来的数据，未必就是"可以随便传播"的数据。

### 2. 免费公开 ≠ 可以随意用

一个网页让你免费打开看，不意味着你可以自动化抓取它的数据。**目的不同**，许可也不同。

### 3. 一人用 ≠ 商业化用

自己看和公开发布是两个世界。法律对"个人研究"的容忍度远远高于"商业化再发布"。

## 二、关键术语扫盲（按从零开始顺序排）

### API（Application Programming Interface，应用程序接口）

**生活比喻**：餐厅菜单。你能按菜单点菜，但不能进后厨。

**技术定义**：网站/平台公开给程序使用的**受控访问入口**。你按规则发请求，它按规则返回数据。

**两种**：
- **授权型 API**：数据商发给开发者的正规接口（如 Tushare）→ 合规
- **内部 AJAX API**：网站自家前端调用的接口，被用户抓包发现（如东财 push2）→ 灰色

### SDK（Software Development Kit，软件开发工具包）

**生活比喻**：餐厅给你的"VIP 点菜 App"。

**技术定义**：数据商发布的 Python/Java 包，把 API 细节封装好。例：`import tushare as ts`。

### Token / API Key（令牌 / 密钥）

**生活比喻**：VIP 卡号。

**技术定义**：调用 API 时附带的身份字符串。

**红线**：Token 是敏感信息——**不能**：
- 上传到 Git / GitHub
- 发到群里
- 截图发朋友圈
- 写死在公开分享的代码里

**正确做法**：放在 `secrets.yaml` / 环境变量 / `.env` 文件里，加入 `.gitignore`。

### Rate Limit（速率限制）

**例子**：Tushare 规定"每分钟 5 次"、东财每秒最多 1-2 次。

**突破后果**：临时封 IP → 永久封号。

**处理**：`time.sleep(1)` 加在每次请求之间。

### User-Agent / Referer / Cookie（HTTP 请求头三剑客）

**本质**：HTTP 请求里"我是谁"的三个声明字段。

- **User-Agent**：我是什么浏览器 / 脚本
- **Referer**：我从哪个页面过来的
- **Cookie**：我的会话 / 登录凭证

**反爬第一道关**：很多网站检查这些，裸写 Python requests 默认的 UA（`python-requests/2.xx`）**一秒被认出是爬虫**。

**例子**（REQ-171 抓天天基金用过）：
```python
HEADERS = {
    "User-Agent": "Mozilla/5.0 Chrome/124.0",
    "Referer": "https://fundf10.eastmoney.com/",
}
```

### 反爬（Anti-Crawler / Anti-Scraping）

**定义**：网站阻止自动化抓取的手段。从弱到强：

| 等级 | 手段 | 应对难度 |
|-----|------|---------|
| 1 | 检查 User-Agent | 简单，加 UA |
| 2 | robots.txt 声明禁区 | 道义约束，非技术 |
| 3 | 请求频率限制 | 加 sleep |
| 4 | IP 封禁 | 换 IP / 降频 |
| 5 | 验证码（CAPTCHA） | 难 |
| 6 | JavaScript 挑战（页面里埋 JS，不执行拿不到数据） | 需浏览器模拟 |
| 7 | AJAX 签名（请求参数要计算哈希，算不对直接 403） | 需逆向 |
| 8 | 行为分析（鼠标轨迹、设备指纹） | 几乎不可能 |
| 9 | 法律手段（起诉） | 商业化越界时启用 |

对照本项目用过的源：
- 天天基金 `jbgk_<code>.html`：1 级（加 UA 就行）
- 东财 push2：3-4 级（限流）
- 雪球：7 级（sign 签名）
- 韭圈儿：7-8 级（App 内嵌 + sign）

### SPA（Single Page Application，单页应用）

**判别**：右键"查看网页源代码"，如果页面主体就是 `<div id="app"></div>` 几行，没有真正的内容，就是 SPA。

**例子**：中证指数官网、雪球、东财 Web 都是 SPA。

**后果**：直接 `requests.get(url).text` 拿到的是空壳，数据不在 HTML 里。

**应对**：
- **逆向 AJAX**：F12 → Network 找真实数据接口（灰色但可行）
- **浏览器模拟**：Playwright / Selenium，让真浏览器跑一遍再取 DOM（重量级）

### AJAX API

SPA 背后的真实数据接口。`F12 → Network → XHR/Fetch` 面板看到的那些。

**使用它就像借着网站的前端脚本"搭便车"**——技术上可行，合规上视 ToS 而定。

### ToS（Terms of Service，服务条款）

每个注册账号时那段"我已同意"的法律文本。

**和数据抓取相关的典型条款**：
- "禁止使用任何自动化手段采集本站数据"
- "禁止将本站数据用于商业目的"
- "禁止将本站数据再发布或转售"
- "本站保留起诉违规用户的权利"

**注意**：
- ToS 不是刑法，但是**民事诉讼的重要依据**
- 违反 ToS 的最直接后果是**封号** + 可能的警告函
- 升级到司法程序的前提：你造成了实际损失或商业化

### robots.txt

网站根目录的文本文件，声明"爬虫规范"。

**查看**：`https://任何网站/robots.txt`

**读法**：
```
User-agent: *                  # 所有爬虫
Disallow: /user/               # 禁止爬 /user/
Disallow: /api/private/

User-agent: Googlebot          # 对 Google 单独规定
Allow: /
```

**性质**：君子协议，不强制。但"爬了禁止路径" = 道义上理亏，法院判决时会参考。

### PoC（Proof of Concept，概念验证）

写个 20-30 行的小脚本验证"这条路能不能跑通"。

**目的**：避免花几小时写完整代码，结果发现数据源根本不通。

**最佳实践**：PoC 脚本放独立临时目录（不混入项目），跑通后再决定是否转正式开发。

## 三、风险分层：你的场景在哪一级？

### Level 1：纯个人使用
- 场景：数据只进自己电脑，自己看
- 风险：**极低**，实务中几乎不会追究
- 约束：不要高频请求（不要把服务器打挂）

### Level 2：小圈子分享
- 场景：几个朋友、小博客、私有 GitHub 仓库
- 风险：**低-中**，取决于数据源 ToS 是否禁止"再分发"
- 约束：
  - 注明数据来源
  - 不要把原始抓取数据直接贴出（发**加工后的可视化结果**更安全）

### Level 3：公开展示（博客 / GitHub Pages / 个人作品集）
- 场景：面向互联网公开访问，但无商业化
- 风险：**中**
- 约束：
  - 数据来源清晰标注
  - 数据处理要有增值（纯粹搬运是高风险）
  - 不要抓登录后或付费的数据
  - 尊重原始网站的 ToS

### Level 4：商业化
- 场景：做成 App / SaaS / 订阅服务 / 数据 API / 广告变现
- 风险：**高**
- 约束：
  - 需要正式授权（签协议或购买商业数据）
  - 自建爬虫的数据基本不能商用
  - 建议咨询专业律师

## 四、红线清单

这些**不要做**：

### ❌ 明确违法

1. **破解 App / JS 的签名算法**（涉及反编译、加密保护）
2. **模拟登录 / 批量注册账号绕过限制**（涉嫌《网络安全法》）
3. **爬取他人隐私数据**（手机号、身份证、个人资料）
4. **用爬取数据做商业化**（订阅、广告、SaaS、数据 API 再售）
5. **高频请求打挂对方服务器**（可能触犯《刑法》286 条"破坏计算机信息系统罪"）

### ⚠️ 高风险（强烈不建议）

1. 爬付费墙后面的数据（理杏仁付费内容、雪球+ 数据）
2. 公开发布爬取的原始数据表格（哪怕只是"搬运"）
3. 把真实 Token 写到公开代码库
4. 批量抓取登录后的个人账户数据

### 🟡 灰色（视场景判断）

1. 爬公开页面做**个人研究**（Level 1）：基本安全
2. 博客里引用一两张图表 + 注明来源（Level 3）：通常可接受
3. 用 AJAX API 做**低频访问**：ToS 可能禁，但**实务容忍度高**

### ✅ 完全安全

1. 使用**授权数据源**（Tushare / 聚宽 / AKShare 的合规接口）
2. 访问**完全公开、无 ToS 禁止**的页面做研究
3. 引用**新闻类报道 + 明显注明出处**

## 五、决策流程：接入新数据源前走这个清单

```
[新数据源进入视野]
        │
        ▼
① 有授权 API 吗？
   ├─ 有 → 阅读 ToS，确认你的场景在允许范围 → 走授权路径（注册 / 付费）
   └─ 无 → ↓
        │
② 目标网站的 robots.txt 允许吗？
   ├─ 明确 Disallow → 放弃或换路径
   └─ 允许 / 未声明 → ↓
        │
③ ToS 是否明确禁止"自动化抓取"？
   ├─ 明确禁止 → 评估你的项目级别
   │    ├─ Level 1-2 → 可谨慎使用，低频 + 个人自用
   │    └─ Level 3-4 → 不要做
   └─ 未明确 → ↓
        │
④ 反爬等级多高？
   ├─ 1-3 级（UA / 限流）→ 加 sleep、伪装 UA 即可
   ├─ 4-6 级（IP 封禁 / JS 挑战）→ 需要 Playwright 等，成本高
   └─ 7-9 级（签名 / 行为分析）→ 放弃
        │
⑤ 值得做吗？
   → 数据价值 vs 维护成本 vs 合规风险 → 综合决策
```

## 六、自检清单（每次重大改动前问自己）

- [ ] 有没有把真实 token / 密码写进会提交的文件？
- [ ] 新增的抓取代码有 `time.sleep()` 吗？
- [ ] 新数据源的 ToS 读过了吗？
- [ ] 页面展示有没有注明数据来源？
- [ ] 有没有把抓到的原始数据直接公开（而不是加工后）？
- [ ] 有没有无意暴露个人隐私信息？
- [ ] 有没有写破解签名、模拟登录这类越界代码？

## 七、FAQ

### Q: 我爬个数据自己看，会被抓吗？
A: 基本不会。个人自用 + 低频 + 不传播 = Level 1，实务中几乎没有追究案例。

### Q: 我在个人博客展示"自己处理过的数据分析图表"，算商业化吗？
A: 通常不算。商业化一般指"直接收钱"（订阅、广告、打赏门槛）。个人展示 + 注明来源 + 无商业收益 = Level 3 合规。

### Q: Tushare token 泄露了会怎样？
A: 别人可以用你的 token 调用 API，消耗你的积分 / 额度。严重时封号。**别截图、别提交公开仓库**。

### Q: AKShare 用的数据也是爬来的，那我用 AKShare 合规吗？
A: AKShare 作为开源聚合库，**它的 MIT 协议允许你使用代码**，但它**封装的上游数据源的合规性** 由你**自己承担**。例如 AKShare 从东财拉的数据，你用它就等于你在用东财的数据，要遵守东财的 ToS。

### Q: GitHub Pages 公开展示数据是 Level 3 吗？
A: 是。公开可访问 + 无商业化 = Level 3。常规约束足以（注明来源、不发原始数据表、不抓登录后数据）。

### Q: 万一数据源关了或 ToS 改了怎么办？
A: 架构层面做降级：失败不阻断主流程、多源备份、`DATA_SOURCES_LANDSCAPE`（本知识库 01 号文档）记录替代源。

### Q: 我发现某个 App 的数据很好用，但 ToS 明确禁止爬取，怎么办？
A: 三条路：① 购买 App 对应的专业版（如有）② 放弃这个源，找替代 ③ 只人工看，不用程序。不要"偷着爬"——违反 ToS 的风险个人承担。

### Q: robots.txt 没禁我就能爬吗？
A: robots.txt 不是"授权书"，只是**君子协议**。它不禁止不等于允许。最终要看 ToS 和实际场景。
