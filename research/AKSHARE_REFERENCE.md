## 这份清单怎么用

这不是“看到接口就马上接”的待办表，而是给 `etf-report` 写脚本时的**候选雷达**。

阅读方式：
- **高置信候选**：已经能对应到明确的官方接口名或官方栏目页，可直接进入技术评估
- **栏目级候选**：已经确认官方文档有对应栏目，但具体接口名需要在真正接入前再二次核验

核心原则：
- **优先补结构化辅助源**
- **先做 source 模块，再让主流程消费**
- **接入前必须看官方文档 + 跑真实数据 + 补测试**
- **当现有主源有数据维度缺口时**（如新浪 getKLineData 缺 `amount`），也可以主动评估主行情链接入（见本技能 REQ-162）

- **适合场景**:

- **已在项目中的定位**:

- **结论**:
  - 这是当前 `etf-report` 与 `AKShare` 最成功的一条接入范式，后续新接口优先复用这套“source 模块 + 标准 JSON + 主流程消费”的设计。

### 2. `fund_name_em` —— 基金基础信息总表入口

- **官方文档页**: `https://akshare.akfamily.xyz/data/fund/fund_public.html`
- **官方文档搜索锚点**: 文档搜索结果已明确出现 `接口: fund_name_em`
- **适合场景**:
  - ETF / 基金代码与名称校验
  - 新 ETF 加入池子时做基础信息核对
  - 生成更稳定的基金别名 / 展示名映射
- **适合本项目的位置**:
  - 作为 `config/config.yaml` 的辅助校验源
  - 可单独生成一份基金基础信息缓存 JSON，不直接塞进主流程
- **接入建议**:
  - 只把它当“元数据校验 / 补全源”，不要替代当前人工确认过的 ETF 池配置

### 3. `stock_zh_index_spot_em` —— 指数实时行情（东财）

- **官方文档页**: `https://akshare.akfamily.xyz/data/index/index.html`
- **官方文档搜索锚点**: 文档搜索结果已明确出现 `接口: stock_zh_index_spot_em`
- **适合场景**:
  - 沪深 300 等基准指数的实时 / 当日快照校验
  - 新浪指数链路异常时的备用参考
  - 主行情链与备用源之间做交叉检查
- **适合本项目的位置**:
  - 先作为“备用源 / 校验源”而不是主替代
  - 可用于健康检查或调试脚本，不建议直接替换当前稳定链路
- **接入建议**:
  - 先做只读比对脚本，确认字段与时间口径稳定后，再讨论是否进入正式流程

## 四、建议的接入顺序

### 第一梯队：最值得优先做

1. **基金流向 / 资金流向**
   - 原因：`fund_flow_data.json` 已预留，项目天然有落点
   - 目标：补出可独立生成的数据文件，供页面解释层或图表层消费

2. **指数备用源 / 校验源**
   - 原因：风险收益比高，不影响主链，先做辅助校验最稳
   - 目标：给基准指数链路增加第二事实源

### 第二梯队：适合增强说明层

3. **基金基础信息 / 档案元数据**
   - 原因：实现成本相对可控，能减少静态配置维护压力
   - 目标：自动校验名称、代码、可能的补充说明字段

4. **宏观补充数据**
   - 原因：更适合做报告解释层升级，而不是底层抓数改造
   - 目标：给宏观分析段落增加客观数据锚点

### 第三梯队：谨慎推进

5. **成分股自动补充 / 自动维护持仓**
   - 原因：收益高，但口径、时效、稳定性要求也高
   - 目标：先做辅助校验，不要直接替代 `holdings.yaml`

## 五、每个候选接口进入开发前都要过的门槛

- **门槛 1**: 能在官方文档里找到明确说明
- **门槛 2**: 参数和返回字段足够稳定
- **门槛 3**: 能独立写成 source 模块
- **门槛 4**: 能输出标准 JSON，而不是把原始 DataFrame 直接传下去
- **门槛 5**: 有空数据兜底、异常降级和最小测试覆盖

如果做不到这 5 条，就先不要进主流程。

## 六、对 `etf-report` 的现实建议

如果下一步真的要做，我建议优先顺序是：

1. **先补基金流向候选调研**：把 `fund_flow_data.json` 变成真正产物  
2. **再补指数备用源比对脚本**：不替换主链，只做交叉校验  
3. **最后再考虑宏观和基金档案补充**：放到解释层，不打断当前日报稳定性

## `etf-report` 写脚本时的推荐查找路径

### 场景 1：已经知道业务主题，不知道接口名

推荐顺序：
1. 打开首页：`https://akshare.akfamily.xyz/index.html`
2. 进入最接近的主题栏目（基金 / 指数 / 宏观 / 股票等）
3. 在栏目页里找接口说明、输入参数、输出字段、示例
4. 确认返回字段是否足够支持项目需求
5. 再决定是否接入 `etf-report`

### 场景 2：已经知道疑似接口名

推荐顺序：
1. 先在官网文档中定位接口页
2. 重点看 4 件事：
   - 输入参数是否可控
   - 输出列名是否稳定
   - 日期字段格式是否清楚
   - 是否支持一次拉全量 / 按时间窗口过滤
3. 通过后再写脚本和测试

### 场景 3：要给现有链路补充结构化事件

优先考虑 `AKShare`，因为它更适合：
- 提供**结构化表格数据**
- 明确列出字段语义
- 作为“辅助事件流”并入主流程

如果是下面这些场景，`AKShare` 往往比直接抓网页更合适：
- 基金拆分 / 份额折算
- 指数或基金补充元数据
- 可结构化的统计类 / 排名类数据
- 适合按 DataFrame 处理的中间层数据

## 当前已落地接口：`fund_cf_em`

### 适用场景

用于识别基金拆分 / 份额折算等会影响历史价格口径的事件，供数据清洗主流程消费。

### 当前项目中的接入方式

`scripts/corporate_action_source.py` 当前使用：
- 接口：`akshare.fund_cf_em()`
- 关键输入：`year`、`typ`、`rank`、`sort`、`page`
- 关键输出列：
  - `基金代码`
  - `基金简称`
  - `拆分折算日`
  - `拆分类型`
  - `拆分折算`

### 在项目里的标准化结果

项目不会把原始字段直接传到业务层，而是统一转换成：
- `action`
- `ex_date`
- `ratio`
- `raw_type`
- `source`
- `note`

这样做的好处是：
- 上游字段名改动更容易集中修复
- 主流程只消费标准结构，不依赖中文列名
- 测试更容易写，也更容易 mock

---

## 推荐脚本模式

### 1. 先调用接口，再立即判空

```python
import akshare as ak

df = ak.fund_cf_em(year="2026")
if df is None or df.empty:
    return []
```

### 2. 把文档列名尽快映射成项目内部字段

```python
row = {
    "code": str(raw["基金代码"]).zfill(6),
    "ex_date": str(raw["拆分折算日"])[:10],
    "ratio": float(raw["拆分折算"]),
}
```

### 3. 在进入主流程前做过滤和标准化

建议至少处理：
- ETF 白名单过滤
- 日期窗口过滤
- 类型映射
- 数值合法性过滤
- 去重

### 4. 保存结构化产物，便于审计和回放

当前项目已经采用这种方式：
- 输出文件：`data/corporate_action_events.json`
- 目的：便于审计、调试、回放与回退

## 接入 `AKShare` 新接口时的检查清单

### 接口选择前
- 这个接口是否来自官网文档而不是二手文章
- 参数和字段说明是否完整
- 是否能稳定表达我们需要的业务语义
- 是否需要分页、按年抓取、或二次筛选

### 编码时
- 不要把中文原始列名散落到多个脚本里
- 统一在一个 source 模块里做原始字段映射
- 记录 `source` 字段，便于审计
- 遇到异常时要能优雅降级，不能把主流程直接拖死

### 验证时
- 覆盖空结果
- 覆盖异常字段 / 非法日期 / 非法数值
- 覆盖窗口过滤
- 覆盖重复记录去重
- 覆盖主流程消费结果是否正确

## `etf-report` 中的使用边界建议

### 适合优先考虑 `AKShare` 的场景
- 结构化基金事件
- 指数 / 基金补充元数据
- 排名、统计、清单型数据
- 适合单独生成 JSON 供主流程消费的数据

### 不建议轻易替换为 `AKShare` 的场景
- 对时效和一致性高度敏感、但当前链路已经稳定的关键主数据（在没有需求缺口的前提下不折腾）
- 已有稳定来源且替换成本高的实时接口

原则上：
- **当现有源有数据维度缺口时**（如本次新浪 getKLineData 缺 `amount`），主动评估 AKShare 接入
- **AKShare 也非常适合作为结构化辅助源**
- **主行情源替换必须单独评估（见 REQ-162），不顺手就改**，但也不是"一律不换"

---

## 常见坑位

- **接口文档是入口，不是最终契约**：上线前一定要跑真实数据验证
- **列名多为中文**：务必在 source 层统一映射
- **日期字段要先标准化**：统一到 `YYYY-MM-DD`
- **数值字段要显式转型**：尤其是比例、金额、成交量
- **空表要有兜底**：不能把空 DataFrame 直接带进主流程
- **保留原始语义**：如 `raw_type` 这类字段在排查时很有价值

---

## 给未来写脚本时的最短行动建议

遇到新的外部数据需求时，按下面顺序做：
1. 先去 `AKShare` 首页找主题入口
2. 找到具体接口页，看参数 / 字段 / 示例
3. 判断它是主数据源还是辅助结构化源
4. 优先写成独立 source 模块
5. 输出标准 JSON，再让主流程消费
6. 补测试，不让业务逻辑直接依赖原始列名
