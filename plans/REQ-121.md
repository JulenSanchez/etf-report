## 动机

当前页面工作流里最核心的结构性问题，不是样式细节，而是**HTML 主文件的职责没有收敛**。

现状已经确认：
- `update_report.py` 读取并写回的是根目录 `index.html`
- `deployer.py` 发布到 GitHub Pages 的也是根目录 `index.html`
- 但 `transaction.py`、`health_check.py`、`print_summary()` 等辅助链路仍把 `outputs/index.html` 当成关键输出文件

这带来三个直接问题：
- **主流程与辅助流程不一致**：真正发布的文件和被检查/备份的文件不是同一个
- **回滚保护不完整**：事务备份当前重点保护的是 `outputs/index.html`，而不是实际被更新的主 HTML
- **阻塞后续能力**：开发调试版、预览流程重构、职责澄清都会被这层路径分裂拖住

因此需要先把“谁是唯一权威 HTML 文件”说死，并把历史残留路径清理干净。

## 目标

以**根目录 `index.html` 为当前唯一权威 HTML 文件**为前提，系统性移除 `outputs/index.html` 在现有主流程中的职责，使更新、校验、备份、预览、发布都围绕同一条主线运行。

## 非目标

- 不在本需求中推进完整的 HTML/JS/CSS 彻底拆分
- 不要求一次性重构全部前端数据注入方式
- 不直接实现开发调试版能力（那是 `REQ-120`）

## 实施步骤

### Step 1: 盘点所有路径引用
- 完整列出 `outputs/index.html` 在脚本、文档、摘要、检查项中的引用点
- 区分“真实主流程依赖”与“历史遗留提示文本”

### Step 2: 明确唯一权威路径
- 正式确认根目录 `index.html` 为当前唯一权威 HTML 文件
- 为预览、校验、发布、备份统一使用同一路径定义

### Step 3: 清理脚本链路
- 修改 `transaction.py`，让备份/恢复保护真实主文件
- 修改 `health_check.py`，让检查对象与真实发布源一致
- 修改 `update_report.py` 的输出验证与总结信息，移除对 `outputs/index.html` 的误导性描述
- 清理对不存在的 `outputs/js/main.js` / `outputs/index.html` 的耦合假设

### Step 4: 清理文档与流程描述
- 更新 `WORKFLOW.md`、`DESIGN.md`、审计/操作说明中的旧路径表述
- 明确 `outputs/` 目录在当前版本中还承担什么，不再承担什么

### Step 5: 验证收敛结果
- 确认主流程运行后，不再生成、依赖、提示 `outputs/index.html`
- 确认没有脚本因为旧路径缺失而降级失败
- 确认为后续 `REQ-120` 提供了单源派生点

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 主文件一致性 | 审查更新/发布/备份/检查链路 | 全部指向同一权威 HTML |
| 路径清理 | 全局搜索 `outputs/index.html` | 无主流程残留引用 |
| 备份有效性 | 创建备份再恢复 | 能恢复真实被修改的 HTML |
| 健康检查 | 跑检查脚本 | 不再因旧路径假设误报 |
| 调试版前置条件 | 评审 `REQ-120` | 可以基于单源设计调试派生产物 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 某些隐蔽脚本仍依赖旧路径 | 中 | 运行时报错 | 先全局盘点，再逐项清理并回归验证 |
| 用户已有预览习惯依赖旧路径 | 中 | 短期认知切换成本 | 同步更新文档与命令输出，明确新路径 |
| 清理时误伤 `outputs/` 目录其他资产 | 低 | 预览资源异常 | 只针对 `outputs/index.html` 职责下手，不扩大范围 |
