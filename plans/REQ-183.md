# REQ-183: 基于 Tuner 寻找最佳策略组合

> **状态**：in_progress（2026-04-28 立项）
> **前置**：REQ-179（done）
> **后续**：REQ-184 接最佳 preset 到主页面发布链路

## 0. 背景

REQ-179 把 Tuner 工具和因子算法都做扎实了，但**当前 YAML 里的 3 个 preset（value / momentum / tech）的参数仍是 REQ-177 时代的拍脑袋默认值**，并不一定是新算法下的最优。

REQ-183 的目标就是：**用 Tuner 跑出来 1~N 个真正可用的 preset 配置**，每个都有明确的"它是什么风格 / 什么市场表现好 / 适合什么持有节奏"。

## 1. 范围

### in scope

1. 用 Tuner 系统性试参数（权重 / 灵敏度 / 信心区间 / 仓位上限 / 偏好强度等）
2. 找出 1~N 个有代表性的策略组合
3. 把选定 preset 写回 `config/quant_universe.yaml` 并补 description
4. 给每个 preset 写一段"策略说明"（在哪里用、回测表现、适合什么市场、最大回撤上限、持有节奏）

### out of scope

- 修改 Tuner 工具（如果调参中发现工具缺陷，开新 BUG 单独跟）
- 修改因子算法（如果发现算法缺陷，回到 REQ-179 范围内迭代）
- 修改主页面发布链路（属于 REQ-184）

## 2. 候选策略风格（初稿，可调）

| 候选 ID | 风格名 | 心智模型 | 关键参数取向 |
|---------|--------|---------|-------------|
| value | 价值定投型 | 长期持有低估品种，少动作 | 高 F4 权重、宽 F2 死区、宽信心死区、低 disc_step |
| momentum | 景气均衡型 | 兼顾趋势和量能，中等换手 | 均衡权重、中性死区、中等 step |
| tech | 技术追击型 | 快进快出抓异动 | 高 F3 权重、窄 F2 死区、窄信心死区、高 step |
| **defensive**（新候选） | 防守保本型 | 熊市/震荡市少出手 | 高 F1 权重、窄信心死区、低仓位上限 |
| **breakout**（新候选） | 主升浪追击 | 在量能验证后重仓 | F3 主导、宽 F1、超宽信心 full_zone |

> 不一定全做，可能合并成 3 个 / 也可能扩到 5 个。最终数量看回测后哪些组合有显著差异。

## 3. 验收标准

每个交付的 preset 必须满足：

- [ ] 在过去 1 年回测下，年化收益 ≥ 15%（基础门槛，多数 ETF 都做得到）
- [ ] Sharpe ≥ 1.0（风险调整后收益合理）
- [ ] 最大回撤 ≤ 25%（不爆仓）
- [ ] 年化收益 / max_drawdown 的 Calmar ≥ 1.5
- [ ] 跟另外几个 preset 在某个维度上有**显著差异**（比如 tech 必须比 value 换手更频繁，defensive 必须比 momentum 回撤更小）
- [ ] 在过去 2 年和 3 年回测下也别崩盘（不要过拟合 1 年）

每个 preset 必须有：

- [ ] description 字段写清楚"什么人用 / 什么场景用"
- [ ] 在 `plans/REQ-183.md` 里有一段策略说明（如下模板）

## 4. 策略说明模板

```markdown
### Preset: <ID>

- **风格**：一句话描述
- **回测表现（1y / 2y / 3y）**：
  - 年化 / Sharpe / 最大回撤 / Calmar
- **适用场景**：什么市场环境最划算？
- **不适用场景**：什么市场会扛不住？
- **持有节奏**：平均换手率、Top-N 持仓平均生命周期
- **关键参数取向**：哪几个参数偏离默认值，为什么
```

## 5. 工作流

1. 启动 Tuner（`python scripts/quant_tuner.py`）
2. 对每个候选风格，反复尝试参数组合，挑表现最好的固化下来
3. 在不同时间窗口（近 1y / 2y / 3y）验证健壮性
4. 验证完成的 preset → 写回 YAML（手动 or 用 Tuner 的"保存参数"按钮）
5. 在本文档第 6 节填写策略说明

## 8. 螺旋优化日志

### 第 1 轮 · 2026-04-29 · 权重网格搜索（Phase 1 + KDP）

**搜索范围**：F1~F4 权重 (step=10%)，bias=0 固定，其他参数用 momentum preset 默认值。
回测窗口：2025-04-29 → 2026-04-29 (近 1 年)，206 个合法组合全跑完。

**Layer 1 事件检测**：自研"主升 / 大跌"识别器（涨幅 ≥ 20% / 时长 ≥ 15 交易日 / 中途回撤 ≤ 10% / 量比 ≥ 1.2x），25 ETF 共识别 74 主升 + 80 大跌，三个标杆案例（515880/512400/159206）全部命中。事件清单 → `data/market_events.json`。

**Layer 2 KDP 评分**：参与度 = 平均仓位 / 理想满仓位 (1/6)；规避度 = 1 - 平均仓位 / 理想满仓位。总分 = 0.6×rally + 0.4×crash。Top 30 候选都跑了。

**Top 候选**（三榜联立）：

| 排名 | (w1,w2,w3,w4) | 年化 | Sharpe | MDD | KDP总 | rally | crash | 备注 |
|------|--------------|------|--------|-----|-------|-------|-------|------|
| KDP#1 | (60, 0, 40, 0) | 74.5% | 2.14 | -13.6% | 55.5 | 40.4 | 78.1 | 三榜均衡，rally 第 1 |
| KDP#2 = Sharpe#2 | (50, 0, 50, 0) | 91.3% | 2.65 | -13.6% | 54.7 | 39.0 | 78.2 | KDP/Sharpe 双优 |
| Sharpe#1 | (40, 0, 60, 0) | 92.4% | 2.74 | -13.0% | 52.4 | 36.7 | 76.0 | 年化最高 |

**核心发现（震撼级）**：

1. **趋势完胜价值**（用户假设 1 ✅）：
   - 趋势型 (w1+w3 ≥ 50, n=121) 平均年化 41.25% / Sharpe 1.53
   - 价值型 (w4 ≥ 30, n=97) 平均年化 12.79% / Sharpe 0.65
   - **3 倍差距**

2. **F4 估值因子在过去 1 年是负贡献**：
   - F4=0 组合 (n=36) 平均年化 63.06% / Sharpe 2.11
   - F4>0 组合 (n=170) 平均年化 23.33% / Sharpe 1.00
   - Top 30 中无任何高 F4 权重组合
   - Bottom 5 全是 w4=60

3. **F2 RSI 异动几乎不增加价值**：
   - Top 10 中 6 个组合 F2=0；其他 F2 也只是 10-30%
   - 加 F2 不能显著改善 KDP 也不能改善 Sharpe
   - 暗示 F2 算法可能有问题（详见第 9 节模型反思）

4. **越能吃主升 ≠ 年化最高**：
   - (60, 0, 40, 0) rally 第 1 (40.4)，年化 74.5%
   - (40, 0, 60, 0) rally 第 9 (36.7)，年化 92.4%
   - 高年化更靠"避坑 + 分散复利"，而不是"集中击穿单波"

5. **信心函数下一步要不要调？暂缓**：
   - 用户决定不调信心函数，转而进入"模型自省"阶段
   - 理由：当前 F2/F4 既没用也说不清"什么时候有用"，应该先修因子设计，再做参数调优

**待解决的疑问**：
- F2 / F4 失效的根本原因？（数据窗口太短？算法假设错了？因子相关性问题？）
- 如何让 F2/F4 至少能"自我解释"它们的有效条件？
- 是否需要拉长回测窗口到 2-3 年验证？

**下一轮目标**：模型自省 → F2/F4 改造 → 重新跑 Phase 1 验证

### 第 4 轮 · 2026-05-06 · 双重复权 bug 修复 + 真实基线重建

**发现问题**：修复前复权数据源后，用户在 Tuner 中发现 2026-01-30→02-02 净值仍有 +24% 虚假跳变。

**根因**：`data_cleaning.py` 对份额拆分日前的价格执行 `price / ratio` 调整。但数据源已切换为前复权(qfq)，qfq本身已包含拆分调整，导致**双重复权**。以 515880（通信ETF，3:1拆分）为例：
- qfq收盘价 Jan29 = 1.054，data_cleaning 再除以3 = 0.351
- qfq收盘价 Feb02 = 1.053（拆分日后，不再除以3）
- 价格从 0.351→1.053，制造了 +200% 虚假涨幅，NAV 从 167→208

**修复措施**：在 `quant_tuner.py` 的 `preload()` 中注释掉 `data_cleaning` 调用——qfq数据不需要手动份额拆分调整。同时清除3处DEBUG代码。

**修复后验证**：
- 2024-08-09→08-12：NAV 92.40→92.02（-0.38%），正常
- 2026-01-30→02-02：NAV 158.95→151.29（-7.66%），正常（<10%涨跌停）
