## 背景与动机

### 当前视图的定位

截至 2026-04-21，日 K / 周 K 主图 Y 轴已经做了两层修正：

1. **路径 B 归一化**：Y 轴从"绝对价格"→"相对 base 的 %"
2. **REQ-164 动态 rebase**：base 从"数据集首根"→"当前可见窗口最左根"

但两层都还是 **线性 Y 轴**。线性坐标下，**同一 % 涨跌在 Y 轴上占用的距离与起点位置有关**：

- 2 元 → 4 元（+100%），Y 轴跨度 = 2 单位
- 100 元 → 200 元（+100%），Y 轴跨度 = 100 单位

即便路径 B 归一化之后 Y 轴单位变成 %，**刻度间距仍然是线性均分**——这对单标的短期视图影响不大，但对"跨量级对比"或"长期观察"存在天然的视觉压缩。

### 对数 Y 轴解决什么

对数轴下，**相同百分比涨跌对应相同 Y 轴长度**：

- 2→4 和 100→200 都是 +100%，在 log Y 轴上占用**相同**的垂直距离
- 2→2.1 和 100→105 都是 +5%，垂直距离也相同

这对 ETF 报告的实际价值在于：

1. **长期视图**：当 dataZoom 拉到全历史（2 年+）时，牛熊周期里不同起点的涨跌幅能被公平比较
2. **benchmark 对比**：ETF 和基准指数可能处于不同价格量级，对数轴下走势陡峭程度更客观
3. **教育意义**：对数轴是"真实收益视角"，配合本技能"风险雷达"的认知视角训练，长期有价值

### 主流炒股软件的现状

同花顺 / 东财 / 富途 / 雪球 默认都用**线性轴**。对数轴通常：
- 藏在图表右上角的设置菜单里
- 需要点击"切换坐标系"
- 大部分散户从未启用过

这也说明：对数轴对普通用户是"高阶功能"，不适合作默认视图——它和 ETF 报告"直观可读"的定位有一定张力。

## 核心设计分歧点（未拍板）

以下是本 REQ 未来实施时必须先决策的 4 个关键点。

### 分歧 1：与当前视图的关系 —— 替换？共存？切换？

| 选项 | 描述 | 适合哪类用户 |
|---|---|---|
| A. **完全替换**默认视图为对数 | 路径 B 归一化 + 对数直接合体 | 量化用户、追求数学正确性的用户 |
| B. **并存**，工具栏加 toggle 按钮（`线性 ⇄ 对数`） | 默认线性，点按钮切换对数 | 兼顾新手与进阶 |
| C. **仅对长期视图启用**（dataZoom 覆盖 > 1 年时自动切对数） | 根据可见窗口长度自适应 | 让系统帮用户决策 |

**建议**：倾向 **选项 B**。
- 理由 1：本技能定位是"投资报告"，不是"量化终端"，默认保持直观
- 理由 2：选项 C 的自动切换可能让用户困惑（"为什么我拉远之后柱子形状变了？"）
- 理由 3：选项 B 的 toggle 按钮可以复用当前调试工具栏的胶囊按钮样式（REQ-141 抽出的图标化规范）

### 分歧 2：对数下"Y 轴刻度值"的语义

路径 B 归一化后，Y 轴值语义是"%"。切到对数后，有两种表达：

| 选项 | 刻度显示 | 含义 |
|---|---|---|
| α. **对数 %**：对 `(price/base - 1) * 100 + 100` 取对数（因为 log 不能处理负数） | `-50% / 0% / +100% / +200%` | 和当前 % 语义一致，但对数只对涨跌幅取对数 |
| β. **对数价格**：Y 轴退回到绝对价格，对价格本身取对数 | `2.00 / 2.50 / 3.00 / 4.00`（刻度不等距） | 抛弃归一化，回到原始价格视图 |
| γ. **混合**：对数坐标但 formatter 仍显示 %（计算 base=价格、pct=(v/base-1)*100，显示 pct） | `-50% / 0% / +100%` | UI 看起来是 %，内部是对数价格 |

**建议**：倾向 **选项 γ**。
- 理由 1：用户已经适应了当前"Y 轴读数就是涨跌幅 %"
- 理由 2：对数价格在 ECharts 里用 `yAxis.type: 'log'` 一个参数就搞定，数据层不用改
- 理由 3：保留路径 B 的归一化心智模型（每次 rebase 都是从 0% 起），同时 silently 把 Y 轴底层变对数

### 分歧 3：负值处理

对数函数不接受负数。但归一化后的 K 线数据可能包含负值（当某根柱子收盘价 < base 时）。

三种处理：

| 选项 | 做法 |
|---|---|
| ① 把 `yAxis.type: 'log'` 作用于**价格**而非 %（选项 γ 路线），天然规避负数 |
| ② 如果走 "对数 %" 路线（选项 α），需要用公式 `y = log(价格 / base)` 直接画对数价格，再 formatter 显示 % |
| ③ 用 `symlog`（对称对数，ECharts 5.3+ 支持 `type: 'log' + logBase` 但不直接支持 symlog，需要自定义 scale） |

**建议**：走选项 ①，即选项 γ 的路线 —— 底层对数作用于价格，UI 层显示 %。这样天然避开负数，不需要额外数学技巧。

### 分歧 4：MA 均线在对数轴下的含义

MA 均线当前是"价格均值"。在对数轴上，算术均值不再是视觉上的"中心线"，几何均值才是。

| 选项 | 含义 |
|---|---|
| i. **保持算术 MA**，直接画到对数轴上 | 视觉上 MA 线会偏离"中值" |
| ii. **切换为几何 MA**（log scale 时）| 数学上更正确，但用户对几何均值的直觉很弱 |
| iii. **对数轴时隐藏 MA** | 简化，避免解释 |

**建议**：倾向选项 i。
- 理由：大多数炒股软件的对数轴模式也是用算术 MA，用户预期相符
- 理由：几何 MA 需要额外计算且解释成本高

## 实现路线图（假设选 B + γ + ① + i）

### 改动点

1. **工具栏（`debug-toolbar.js` 或单独的 `chart-toolbar.js`）**
   - 在每个 K 线图的右上角（或 legend 旁）加一个胶囊按钮：`📈 线性` / `📉 对数`
   - localStorage 持久化偏好（`kline-y-scale-mode: linear | log`）
   - 点击时触发图表重渲染

2. **`report-main.js::renderDailyChart` / `renderWeeklyChart`**
   - 新增参数 `yScaleMode = 'linear' | 'log'`
   - `yAxis[0].type` 依此设为 `'value'` 或 `'log'`
   - **关键**：对数模式下，K 线 data 不能是路径 B 的"归一化 %"，因为 log 不接受负数；需要回退到"原始价格"作为 data，然后 Y 轴 formatter 改成 `v => ((v/baseOpen - 1) * 100).toFixed(2) + '%'`（选项 γ）
   - tooltip 逻辑保持不变（仍显示原价 + 相对昨日/上周）

3. **REQ-164 动态 rebase 兼容**
   - 对数模式下，base 仍然跟随 dataZoom，但 `buildNormalizedSeriesData` 输出**原始价格**（不归一化），由 Y 轴 formatter 负责 % 显示
   - 等价于：对数模式下"归一化"只在视觉层发生，数据层是原始价格

4. **benchmark 对比线**
   - 也用原始价格（需要从 `klineData[code].benchmark.closes` 取）
   - Y 轴共享后 formatter 会自动帮它也显示 %

5. **测试**
   - JS 侧目前没测试框架，手动 smoke 为主
   - Python 侧无影响

### 改动风险

- **Y 轴刻度间距变陡**：对数下，涨幅大的柱子之间距离会变大；第一次看到可能觉得"图变怪了"——建议 toggle 按钮 tooltip 写清楚"价格坐标按涨跌幅等距显示"
- **与 splitLine 交互**：对数下的 splitLine 刻度间距不均，视觉上可能出现密集区，需要测下效果
- **dataZoom 的 start/end 锚点**：对数轴下 dataZoom 的行为和线性轴理论上一致（ECharts 内部处理），但建议实测
