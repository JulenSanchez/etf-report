## 2. v4.0.0 待建设

### 2.1 人设仪表盘（Persona Dashboard）— 第三大页面

**决策（2026-06-03）**：独立页面 `dashboard.html`，纯静态（预计算 JSON payload）。与 `index.html`（正式报告）、`tuner.html`（工坊）平级。

v4 的核心交付：不是"三列参数对比"，而是**三个人设在各自说话**。底层数据同一份，但组织逻辑、强调指标、解读口径完全不同。

**信息架构**：

```
dashboard.html
├─ 三 Tab 切换器（精算师 / 禅修者 / 赌徒）
│
├─ [每个 Tab 内]
│   ├─ 哲学宣言区 → 人设的自我陈述（来自 persona-declarations）
│   ├─ 核心关切卡 → 人设最在意的指标（不是通用指标）
│   │   精算师: S×C 趋势、F7 信号分布、drawdown 热力图、佣金燃烧率
│   │   禅修者: Sharpe 趋势、最大单票仓位稳定性、调仓次数、最长持有期
│   │   赌徒:   AR 趋势、连胜记录、重仓信念度、命中率
│   ├─ 当前状态感知 → 同一份数据，但解读口径不同
│   │   "F7 在 X 板块强劲，在 Y 板块衰弱。我的反转押注在 Z 上正在兑现。"
│   │   vs "市场震荡。我保持不动。Top 3 持仓已 8 周未变。"
│   ├─ 策略健康自评 → 以自身目标函数衡量自己
│   └─ 演化立场 → 当前是否有待投票的提案？本派的立场和理由
│
├─ 三派对比雷达图（因子暴露 / 扇区偏好 / 风险指标）
└─ 选股分歧日历（哪些天三派选了不同的 ETF）
```

**技术方案**：
- 数据来源：`update_report.py` 管线生成 `assets/js/dashboard_payload.js`
- 渲染：纯前端 `assets/js/dashboard.js` + `assets/css/dashboard.css`
- 不需要 Flask 后端（与 Tuner 解耦）
- 详见 `plans/REQ-270.md`

### 2.2 版本演化协议（软分叉）

**决策（2026-06-03）**：不允许真分叉。多人设必须运行同一个回测引擎版本，仅在参数层面分叉。

规则：
1. **新因子/新规则提案**：AI 或用户提出 → 自动跑三派回测
2. **独立表态**：各自用目标函数判断接受/拒绝
3. **分歧处理**：新因子默认进入引擎；不接受的人设设权重=0（不使用该因子）
4. **分歧升级**：用户/中心 AI 裁决 → 会话内手动决定
5. **版本记录**：记录谁在用哪个版本的哪组参数（策略版本树，但只分参数不分引擎）

详见 `plans/REQ-272.md`

### 2.3 回测工作台（Backtest Workbench）

解决当前性能瓶颈的基础设施升级：

- **共享数据层**：启动时一次性加载 45 支 ETF 全史数据到进程内存
- **多进程调度**：`ProcessPoolExecutor` 并行跑多窗口/多参数回测
- **结果缓存**：已有 `last_backtest.json` → 扩展为键值缓存 `{preset}_{window}_{param_hash}`
- **Queue 调度**：TPE 优化 → 采样队列 → 多 worker 并行跑 → 结果批量回传
- **性能目标**：40 trial × 2Y TPE 从 30min → 10min（3x 加速）

详见 `plans/REQ-271.md`

详见 `plans/REQ-273.md`

### 2.5 多 Agent 动态（v4 远期愿景）

**决策（2026-06-03）**：v4 的"多主体"不只是一个页面展示概念，而是架构上的 Persona 主体化。即使底层数据/代码与 Tuner 相同，v4 仪表盘的核心区别在于 **"视角"**——每个人设对策略运行状态有主人翁式的感知。

v4 的静态仪表盘是第一步。远期每个 Persona 作为独立 Agent：
- **自主监控**：定期检测自己的等权目标函数是否退化
- **自主提案**：退化时自动触发 TPE 参数搜索
- **自主投票**：框架级变更（新因子/新执行逻辑）时各自用目标函数投票
- **自主分叉**：策略版本树，参数层面分叉（引擎版本统一）

这本质上是把当前手工的"发现问题→跑 TPE→人工判断→落地"流程自动化。
