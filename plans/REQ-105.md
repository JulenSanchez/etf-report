## 动机

当前日志记录的是"发生了什么"（INFO/WARN/ERROR），但没有系统性地追踪"每步操作耗时多少、输入输出是什么"。排查问题时需要翻大量日志拼凑链路，效率低。

**现状问题**：
- 不知道某次报告更新各 API 调用分别花了多久
- 数据处理步骤没有输入/输出摘要，出错时难以定位
- 文件操作（HTML 写入、JSON 备份）没有记录

**不做会怎样**：问题排查靠猜。影响：低（系统当前稳定，这不是紧急痛点）。

## 实施步骤

### Step 1: 定义审计事件模型
- 在 `logger.py` 扩展 `audit` 级别（INFO 之上，WARN 之下）
- 事件结构：
  ```json
  {"ts": "...", "level": "AUDIT", "type": "api_call|data_process|file_io",
   "detail": "...", "duration_ms": 123, "status": "ok|error", "extra": {}}
  ```
- **涉及文件**：
  - `scripts/logger.py` — 修改，新增 audit 级别
- **预期产出**：审计日志框架就绪

### Step 2: 埋点 API 调用
- 包装 `requests.get`（装饰器或上下文管理器）
- 记录：端点 URL、响应时间、状态码、返回数据量
- **涉及文件**：
  - `scripts/update_report.py` — K 线 API 调用埋点
  - `scripts/realtime_data_updater.py` — 实时行情 API 调用埋点

### Step 3: 埋点数据处理
- 在 `fix_ma_and_benchmark.py` 各关键函数入口/出口记录摘要
- 输入：ETF 数量、数据条数；输出：计算耗时、异常数
- **涉及文件**：
  - `scripts/fix_ma_and_benchmark.py` — 修改，添加审计日志

### Step 4: 埋点文件操作 + 审计日志输出
- HTML 写入、JSON 写入/备份时记录文件大小和操作结果
- 审计日志独立输出到 `logs/audit_{date}.jsonl`
- **涉及文件**：
  - `scripts/update_report.py` — 文件操作埋点
  - `scripts/logger.py` — 审计日志独立文件输出

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 审计日志生成 | 跑一次报告后检查 `logs/audit_*.jsonl` | 文件存在且非空 |
| API 调用记录 | 检查审计日志 | 每次请求有 duration_ms 和 status |
| 数据处理记录 | 检查审计日志 | MA 计算等步骤有耗时记录 |
| 文件操作记录 | 检查审计日志 | HTML/JSON 写入有文件大小记录 |
| 回归 | 跑一次完整报告 | 报告正常生成，原有日志不受影响 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 审计日志体积增长 | 中 | 磁盘占用 | 按天轮转，保留最近 7 天 |
| 埋点代码侵入业务逻辑 | 低 | 代码可读性下降 | 用装饰器/上下文管理器封装，业务代码只加一行 |
| 性能影响 | 低 | 报告生成变慢 | 审计记录是写文件，IO 很小，影响可忽略 |

| `tests/test_audit.py` | 新增 | 审计功能测试（7 个） |

> 不修改任何现有业务逻辑，只在关键路径上加审计埋点（一行 with 语句），审计写入失败不阻塞业务。
