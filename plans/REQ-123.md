## 动机

当前 `etf-report` 的状态栏能力已经验证出较强的协作价值，但它的定义分散在 `.mdc` 规则与 `PLAN.md` 说明文里，导致三个问题：

- **不可迁移**：别的工作区或技能难以复用，只能连同现有文档结构一起搬运
- **边界混杂**：需求看板文档承载了不属于需求本身的状态网络内容
- **扩展困难**：难以在同一工作区下并存多个技能级状态网络，也难以给新用户提供统一模板

因此需要把状态栏能力从 `etf-report` 现有语义里抽离，重构为一套可迁移的 **协议 + 配置** 机制：

- **协议** 负责说明状态栏长什么样、配置如何被发现/选中、核心字段代表什么
- **配置** 负责描述某个具体主题下的状态网络，如 `etf-report` 开发状态网络

## 本轮目标

本轮从"草案"继续推进到"首轮落地"，但仍不进入运行时实现：

- 定义协议的职责边界与运行模型
- 定义配置的发现/选中机制
- 定义 v1 核心 schema
- 为 `etf-report` 提供一份能承接现有语义的配置模板方向
- 给出 `PLAN.md` 旧状态网络到新配置结构的迁移映射
- 将 `PLAN.md` 中的旧状态网络正文瘦身为协议入口，减少双份定义
- 通过 `extensions` 机制兼容暂未纳入核心 schema 的专属语义

## 范围边界

### 本轮纳入
- 状态栏三行结构与渲染约束
- 协议激活后的配置发现 / 选中流程

- `extensions` 扩展区的定位
- `etf-report` 作为技能级配置实例的落位方式

### 本轮不纳入
- 入口守卫 / 出口守卫本身的规则设计
- 需求看板文件的治理规则
- 直接改造 `.mdc` 为新运行时实现
- 运行时实现细节与自动迁移脚本

## 实施步骤

### Step 1: 收敛协议职责与运行时模型

- 定义默认配置与内置兜底的优先级

### Step 2: 重构配置模型
- 从"说明文结构"改写为"状态网络配置结构"

- 明确哪些语义进入核心 schema，哪些进入 `extensions`

### Step 3: 产出可评审草案

- 写出配置模板骨架与字段说明

# 附录 A：状态栏协议草案（v1）

> 原文件：`plans/REQ-123-drafts/STATUSBAR_PROTOCOL_DRAFT.md`，2026-04-22 整合至此。

## A.1 目标

把当前 `etf-report` 已验证有效的带状态回复能力，抽离为一套可迁移的 **状态栏协议 + 状态网络配置** 机制，使其能够：

- 在不改造目标工作区业务内容的前提下迁移到其他工作区
- 支持同一工作区下多个主题 / 技能各自拥有独立配置
- 让新用户仅通过模板即可理解如何定义自己的状态网络
- 将 `PLAN.md` 等需求面板文档中的状态网络内容完整迁出，避免边界混杂

## A.2 设计原则

- **单一事实源**：状态栏协议由协议文档定义；某个主题的状态网络由其配置文件定义
- **先发现、后持有**：协议在激活时执行一次配置发现与选中，选中后进入持有阶段
- **配置作用域就近生效**：配置可以放在技能/模块/工作区级目录下，真正决定生效的是配置内部声明的作用域与激活条件
- **v1 核心最小化**：v1 核心 schema 只保留任何工作区都容易理解的概念，专属语义通过 `extensions` 承接

## A.3 协议职责

1. **显示协议**：定义状态栏四段式结构 `{网络段} | {状态段} | {micro-stage} | {上下文}`
2. **发现协议**：定义如何在当前工作区内搜索所有候选配置
3. **选中协议**：定义多个配置同时存在时如何选中活动配置
4. **结构协议**：定义配置文件的最小结构、核心字段含义、降级行为

## A.4 运行模型

- **激活阶段**：协议被触发后，在 `.codebuddy/` 范围内搜索候选配置，结合对话上下文选出活动配置
- **持有阶段**：后续回复默认沿用活动配置，不重复全量搜索
- **重判阶段**：用户切换主题、配置失配、配置更新、用户要求重判时重新发现与选中

## A.5 配置发现与选中

- **候选识别**：需声明 `config_type`、`name`、`protocol`、激活条件
- **选中优先级**：更具体主题 > 显式路径命中 > 更高 priority > 工作区默认 > 协议内置
- **降级行为**：工作区默认配置 → 协议内置配置

## A.6 v1 核心 schema 字段冻结总表

| 区块 | 保留 | 合并 | 删除 |
|------|------|------|------|
| frontmatter | `config_type`, `name`, `protocol`, `scope`, `summary`, `priority` | - | - |
| Activation | `keywords`, `paths`, `topics`, `examples`, `notes` | - | - |
| States | `key`, `emoji`, `summary`, `context_format`, `enter_when`, `nesting`, `transitions`, `notes` | `label` → 使用小节标题 | - |
| Actions | `key`, `emoji`, `summary`, `context_format`, `trigger_when`, `interrupts_state`, `notes` | `label` → 使用小节标题 | - |
| Routing | `rules` | `order` → 用书写顺序表达优先级 | - |
| Stacking | `operations`, `notes` | `default_behavior` → 融入 `notes` | `stacked_types`, `non_stacked_types` |
| Display | `network_marker`, `fallback_context`, `context_presets`, `notes` | - | `header_template`, `separator`, `markdown_safe_mode` |
| Extensions | `extensions` 下任意扩展块 | - | - |

## A.7 下一阶段建议

1. 把协议草案从草案收敛为正式协议文档
2. 在其他工作区验证"仅迁移宿主规则 + 新增配置文件"是否足以跑通
3. 明确运行时如何发现、缓存、重判活动配置

# 附录 B：状态栏配置模板（v1 通用）

> 原文件：`plans/REQ-123-drafts/STATUSBAR_CONFIG_TEMPLATE.md`，2026-04-22 整合至此。

## B.1 最小可用配置要求

| 部分 | 是否必填 | 最低要求 |
|------|----------|----------|
| frontmatter | 必填 | `config_type`、`name`、`protocol`、`scope`、`summary` |
| Activation | 必填 | `keywords` / `paths` / `topics` 三者至少有一种非空 |
| States | 必填 | 至少定义 1 个状态 |
| Routing | 条件必填 | 当存在多个可选状态/动作时必须提供 |
| Display | 条件必填 | 非 `workspace-default` 配置必须提供 `network_marker` |
| Actions / Stacking / Extensions | 可选 | - |

## B.2 空白模板

```markdown
---
config_type: statusbar
name: your-config-name
protocol: statusbar/v1
scope: skill
priority: 50
summary: 用一句话说明这份配置服务于什么主题
---

## Activation
- **keywords**: `至少填一类命中线索`
- **paths**: `可选，路径线索`
- **topics**: 可选，主题说明

## States

### 讨论
- **key**: `discussion`
- **emoji**: `💬`
- **summary**: 描述这个状态是什么意思
- **enter_when**: 什么时候进入这个状态
- **context_format**: 右侧上下文怎么写
- **nesting**: false
- **transitions**: `review`, `bug`

## Actions

### 执行
- **key**: `execute`
- **emoji**: `🔄`
- **summary**: 描述这个动作做什么
- **trigger_when**: 什么时候触发
- **context_format**: 在做什么
- **interrupts_state**: false

## Routing
- **rules**:
  - **match**: `关键词或正则`
    - **target_type**: `state`
    - **target**: `discussion`
    - **reason**: 为什么命中这里

## Display
- **network_marker**: `除 workspace-default 外必填`
- **fallback_context**: `进行中`
- **context_presets**:
  - `discussion`: `讨论主题`
  - `execute`: `在做什么`

## Extensions
### custom_notes
- 放当前 schema 不好表达但又想保留的专属内容
```

## B.3 填写要点

- **`scope`**：`skill` / `module` / `workspace-default`
- **States vs Actions**：State 有持续时间进入上下文；Action 是瞬时操作
- **`network_marker`**：建议 `专属 emoji + 短名`，如 `📊 ETF`、`🧠 Risk`，一旦选定长期保持稳定
- **`micro-stage`**：不是配置字段，而是协议运行时生成的片段级提示（`emoji + prompt`）

## B.4 最小示例

```markdown
---
config_type: statusbar
name: my-feature-dev
protocol: statusbar/v1
scope: skill
summary: 某个功能的开发状态网络
---

## Activation
- **keywords**: `my-feature`

## States
### 讨论
- **key**: `discussion`
- **emoji**: `💬`
- **summary**: 默认讨论状态
- **context_format**: 讨论主题

## Display
- **network_marker**: `🧩 MyFeature`
```
