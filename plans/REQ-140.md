# REQ-140: 按 ID 模糊匹配与多结果逐条定位

## 动机

现有第三个调试工具已经支持按完整 `id` 精确定位，但当用户手里只有类似 `strong-buy-card-overview-item-` 这样的**前缀或片段**时，仍然无法直接命中目标。

对于这类带 `*` 通配含义的元素族，真正需要的是“先搜到一组候选，再逐条切换确认”的交互，而不是只接受单个完整 `id`。

## 目标

在现有按 id 聚焦工具上继续增强，支持：
- 输入完整 `id`、前缀或中间片段进行匹配搜索
- 当命中多条时，默认先聚焦第 1 条
- 提供“上一个 / 下一个”逐条切换结果
- 在输入面板内常驻显示匹配总数与当前序号
- 有结果信息后，去掉定位成功时的临时 toast 提示

## 设计原则

- **兼容旧能力**：完整 `id` 仍应像以前一样可直接定位
- **搜索优先**：把输入框从“精确查询”升级为“匹配搜索”
- **结果常驻**：匹配数量和当前索引在面板里持续可见，而不是闪一下就消失
- **逐条核对**：多结果场景要支持稳定切换，便于逐个确认页面元素
- **弱打扰反馈**：成功态尽量内聚到面板内，减少临时 toast 打断

## 实施步骤

### Step 1: 重构匹配逻辑
- 从 `getElementById` 扩展为支持完整/前缀/片段匹配
- 过滤调试 UI 自身节点，避免误命中

### Step 2: 扩展面板交互
- 增加结果状态区
- 增加“上一个 / 下一个”按钮
- 让按钮禁用态与匹配结果同步

### Step 3: 串联定位与导航
- 首次搜索自动跳到第 1 条
- 上一条/下一条沿当前结果集循环或顺序切换
- 切换时维持自动切页、居中滚动和聚焦高亮

### Step 4: 回归与收口
- 验证 `strong-buy-card-overview-item-` 这类片段搜索
- 验证单结果 / 多结果 / 零结果三种状态
- 验证与现有调试开关、热区模式继续共存

- 2026-04-18: 实现完成。`index.html` 已将原“按 id 聚焦元素”面板升级为搜索态：支持完整 id / 前缀 / 片段三类匹配，按“精确 > 前缀 > 包含”排序结果，默认聚焦第 1 条，并提供“上一个 / 下一个”导航；面板内常驻显示“共匹配 N 条 · 当前 i/N”和当前命中的 id，定位成功时不再额外弹成功 toast。
- 2026-04-18: [用户覆盖] AI 原实现：`搜索并定位` 独占上一排，并额外提供“清除高亮”按钮；用户决定：去掉“清除高亮”，改为关闭聚焦功能时自动清除高亮，同时把蓝色主按钮挪到与“上一个 / 下一个”同排，名称简化为“搜索”；影响：聚焦面板按钮区固定为 `搜索 / 上一个 / 下一个` 三键布局。
- 2026-04-19: 收尾补齐。回读发现“关闭聚焦功能时顺带清除高亮”已写入评审记录，但 `index.html` 的 `setFocusPanelOpen(false)` 尚未真正接入清理逻辑；现已补齐关闭即清高亮，确保最终实现与本需求的 UI 决策一致。
