## 动机

> v4 的核心交付物。当前两大页面（`index.html` 正式报告 + `tuner.html` 工坊）都是"工具视角"——用户操作参数、看结果。v4 需要第三人称视角：三个人设作为独立主体，各自用自己的哲学、关注点和语言来"感知"策略运行状态。

**现状问题**：
- Tuner 可以切换 preset，但切换的是"参数"，不是"视角"
- 三个人设的终局参数已锁定，但它们在代码里只是 YAML 里的三组数字——没有"人格"
- 用户看不到每个人设的"内心活动"：它关心什么？它觉得当前市场怎样？它对策略满意吗？

**不做会怎样**：
- v4 的"多主体策略治理"只是一句口号
- 三个人设始终只是 Tuner 里的三组参数预设，无法真正形成"为谁服务"的差异化

## 核心设计约束（2026-06-03 用户确认）

1. **独立页面** `dashboard.html`，与 `index.html`、`tuner.html` 平级
2. **纯静态**：预计算 JSON payload，由 `update_report.py` 管线生成
3. **不需要 Flask 后端**：与 Tuner 解耦
4. **"视角"优先于"数据"**：底层数据可以跟 Tuner 一样，但每个人设的组织逻辑、强调指标、解读口径完全不同
5. **每个人设对自己的策略有"主人翁感"**

## 实施步骤

### Step 1: 数据层 — payload 生成管线
- 新增 `scripts/build_dashboard_payload.py`
- 由 `update_report.py` 调用，生成 `assets/js/dashboard_payload.js`
- payload 结构：
  ```js
  window.__DASHBOARD__ = {
    generated_at: "2026-06-03",
    personas: {
      actuary:    { philosophy, core_metrics, self_assessment, evolution_stance, ... },
      zen:        { ... },
      gambler:    { ... }
    },
    cross_persona: {
      divergence_calendar: [...],
      radar_comparison: {...},
      evolution_timeline: [...]
    }
  }
  ```
- 涉及文件：`scripts/update_report.py`（新增调用点）、`scripts/quant_backtest.py`（复用回测数据）

### Step 2: 前端 — HTML 骨架 + CSS
- 新建 `dashboard.html`
- 新建 `assets/css/dashboard.css`
- 三 Tab 切换器 + 每个 Section 的 DOM 骨架
- 响应式布局（桌面优先）

### Step 3: 前端 — JS 渲染引擎
- 新建 `assets/js/dashboard.js`
- 读取 `window.__DASHBOARD__`，渲染各 Section
- Tab 切换逻辑
- 雷达图（ECharts 雷达图，复用现有 ECharts CDN）

### Step 4: 人设"内心独白"生成
- 初期用 Python 模板引擎（基于 if/else 规则 + 阈值判断）
- 远期接入 LLM 生成（调用 AI API 生成自然语言解读）
- MVP 阶段用结构化模板即可（伪自然语言但有人设腔调）

### Step 5: 集成测试 + 健康检查
- `health_check.py` 新增 dashboard 检查项
- 验证 payload 生成完整性
- 本地 Edge 打开 `dashboard.html` 验证

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| payload 体积过大 | 中 | 加载慢 | 按 Tab 懒加载，只预加载当前人设数据 |
| 人设"独白"模板化太生硬 | 高 | 用户觉得不够"活" | MVP 接受模板化，v4.1 接入 LLM 生成 |
| 三人设数据一致性漂移 | 低 | 仪表盘显示矛盾数据 | 同一 payload 生成时间戳，数据来自同一次回测 run |
