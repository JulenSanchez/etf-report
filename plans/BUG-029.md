# BUG-029: F1 因子 lookahead — intraday 周线重建污染

**严重度**: 🔴 Critical
**状态**: fixed
**引入版本**: v3.0.0（量化回测大版本，`rebuild_weekly_from_daily` 引入时）
**发现日期**: 2026-06-03
**修复版本**: v3.5.0-dev
**关联**: REQ-265（QDII 停牌感知）
**耗时**: ~8 人时（跨 3 天）

---

## 现象

拉取 6/3 盘中数据后，回测中 6/2 的 F1 因子值发生变化。用户三天内反复遇到，每次盘中操作后回测结果不一致。

## 根因

### 调用链

```
refresh_data() → intraday_cache[code] = {date: "2026-06-03", close: ..., volume: ...}
                    ↓
/api/run (end_date = today)
  ├─ _get_daily_with_cache(code) → 日线 DataFrame APPEND 6/3 行
  ├─ _get_weekly_with_cache(code) → rebuild_weekly_from_daily(daily_with_jun3)
  │     ↓
  │   ISO 周聚合: 第23周 date = "2026-06-03"(最后一天), close = 6/3 盘中价
  │     (重建前: date="2026-06-02", close=6/2 收盘价)
  │     ↓
  └─ _precompute_factors() → F1 EMA 全序列重算
        ↓
      searchsorted(weekly_dates, "2026-06-02", side="right")
        "2026-06-02" < "2026-06-03" → 命中 index=319 (第22周, 5/29)
        (修复前: "2026-06-02" == "2026-06-02" → 命中 index=320 (第23周, 6/2))
```

### 核心机制

不是 EMA 本身用了未来数据。EMA(t) 只依赖 close[0..t]，计算是时间对齐的。

问题是 **`rebuild_weekly_from_daily` 用 ISO 周的最后一天作为 bar 日期**。加了 6/3 后，第 23 周的最后一天从 6/2 滑到 6/3 → `searchsorted("2026-06-02", side="right")` 发现 6/2 < 6/3 → 跳过第 23 周 → 回退到第 22 周。**F1 读了完全不同一周的值。**

## 修复

`quant_tuner.py:1087-1102`：准备 preloaded 数据时，**all_weekly 永远不合并 intraday cache**——始终保持 CSV 快照。all_daily 正常合并（日线价格查询需要）。

```python
# 修复前
if CACHE.get("intraday_cache"):
    for code in CACHE["all_daily"]:
        all_daily[code] = _get_daily_with_cache(code)
        all_weekly[code] = _get_weekly_with_cache(code)  # ← 污染

# 修复后
if CACHE.get("intraday_cache"):
    for code in CACHE["all_daily"]:
        all_daily[code] = _get_daily_with_cache(code)
    # all_weekly 不变 — CSV 快照, 不重建
```

## 为什么难排查

### 1. 症状远离根因
用户看到"F1 变了"，直觉是 EMA 计算或因子映射有 bug。实际是周 bar 的**索引偏移**——searchsorted 命中不同行。

### 2. 跨两层数据
日线 → `rebuild_weekly_from_daily` → 周线 → `searchsorted` → 因子查表。每层单独看都是正确的，组合起来才有问题。

### 3. 只在特定时间窗口触发
只有盘中拉取未来日期数据后，再回测包含"当周前几天"的日期时才触发。盘后 CSV 更新后窗口消失。

### 4. AI 的两次错误分析
- **第一次**：说 F1 每天随周收盘价变化——完全错误。用户纠正：F1 在周内盘中是不变的（CSV 没更新）。
- **第二次**：用 `end_date >= ic_date` 做守卫——逻辑正确但没覆盖真正的场景。回测 `end_date=今天` 时，昨天（6/2）的信号也在同一次回测里，周线仍然被重建污染。

### 5. `side="right"` 的语义陷阱
`searchsorted(v, side="right")` 返回第一个 > v 的索引。当周 bar 的日期等于 v 时，返回 len（末尾之后）；当周 bar 日期大于 v 时，返回该 bar 的位置。同一个 v，因为一个毫不相干的日期的数据加入，导致返回不同的索引。

## 教训

1. **"重建"操作是危险的**：`rebuild_weekly_from_daily` 破坏了 CSV 周线的稳定性。周线数据应该只从已确认的 CSV 读取，不应被实时数据重建。
2. **因子计算与价格查询应使用不同时间粒度的数据源**：日线可以实时，周线必须用已确认的快照。
3. **当 AI 对机制的解释被用户纠正时，应该更彻底地怀疑自己的理解**：第一次纠正（F1 周内不变）已经暗示了 searchsorted 机制和我想的不一样，但我没有深究。
4. **跨层 bug 需要画完整调用链**：从 `refresh_data` → `_get_weekly_with_cache` → `rebuild_weekly_from_daily` → `searchsorted` → `f1_val[idx-1]`，每一步都画出来才能定位。
