# REQ-361: 前后端参数缩放一致性审计

**优先级**: 🔴 P0 | **状态**: backlog | **目标版本**: v3.12.0

## 背景

BUG-043（前沿点极端参数 NaN 崩溃）排查时发现：`c_sensitivity` 和 `concentration` 存在 ÷10 缩放机制——前端 UI 滑块显示 2.8，后端存储/传输是 28，引擎消费时 ÷10 得 2.8。

AI 当时只在 BUG-043 临时解释了缩放，**没有反向追溯缩放链路是否在所有路径上对齐**。BUG-043 的现象（极端参数导致 softmax 溢出 → NaN 崩溃）很可能就是某条路径上缩放不一致导致的——但只修了 `idxmax()` 防护，根因未查。

## 风险

如果缩放在某一条路径上遗漏，会出现：

| 风险场景 | 后果 |
|---------|------|
| YAML 预设存了缩放后值，`preset_to_tuner_params` 不 ÷10 | 前端滑块显示 28（应为 2.8），用户调参混乱 |
| 前端 `getParams()` ×10，`tuner_params_to_config_override` 不 ÷10 | 引擎收到 28（应为 2.8），softmax 溢出 → NaN 崩溃 |
| 种子参数 enqueue 时缩放不一致 | TPE 在错误尺度下采样，收敛慢或搜不到 |
| `_REQUIRED_PARAMS` 校验在缩放前/后不一致 | 合法参数被拒，或非法参数通过 |

## 排查范围

### 1. 列出所有带 scale 的参数

grep `PARAM_SCHEMA` 中所有带 `unit`/`scale` 字段的参数，记录：
- 参数名
- UI 显示范围
- 存储值范围
- 引擎消费值范围
- 缩放系数

### 2. 四条路径的缩放对齐

| 路径 | 函数 | 检查点 |
|------|------|--------|
| UI → 引擎 | `tuner_params_to_config_override()` | 是否对所有带 scale 的参数 ÷10？ |
| 引擎 → UI | `preset_to_tuner_params()` | 是否对所有带 scale 的参数 ×10？ |
| YAML → 种子 | `seed_params_from_presets()` | 从 YAML 读出的参数是否需要缩放？ |
| UI → 种子 | `getParams()` (tuner.html) | 读滑块时是否统一 ×10？ |

### 3. 对称性测试

写一个测试（加入 `tests/test_quant_contract.py`）：

```python
def test_param_scale_round_trip():
    """随机生成 N 组 params，验证 前端→后端→引擎→反向→前端 与原值一致"""
    for _ in range(100):
        original = random_params()
        ui_to_engine = tuner_params_to_config_override(original)
        engine_to_ui = preset_to_tuner_params(ui_to_engine)
        assert engine_to_ui == original  # 缩放对称
```

### 4. YAML 预设存储约定

确认 `config/quant_universe.yaml` 里的 `c_sensitivity`/`concentration` 等参数存的是缩放前还是缩放后值，在 `param-lifecycle.md` 文档化。

### 5. 极端参数测试

用 BUG-043 的极端参数组合（c_sensitivity=28, concentration=26.1）跑回测，验证：
- 修根因前：NaN 崩溃
- 修根因后（缩放对齐）：正常完成，结果在合理范围
