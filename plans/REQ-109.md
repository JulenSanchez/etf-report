## 动机

当前 `index.html` 约 443KB，其中内联 CSS 和 JS 占大部分体积。虽然 GitHub Pages 加载速度尚可，但优化资源体积是好习惯，也能加快 `file://` 本地打开速度。

**现状问题**：
- 内联 CSS/JS 未压缩，有大量空白、注释、冗余声明
- `--publish` 模式下推送到 GitHub Pages 的文件可以更小

**不做会怎样**：无直接影响。影响：低（GitHub Pages 有 CDN，加载速度可接受）。

## 实施步骤

### Step 1: 分析当前资源体积
- 统计 index.html 中 `<style>` 和 `<script>` 的原始大小
- 评估压缩潜力
- **涉及文件**：无代码改动
- **预期产出**：体积基线数据

### Step 2: 接入压缩工具
- 选用 Python 原生库（零 Node 依赖）：
  - CSS: `csscompressor`（pip install csscompressor）
  - JS: `rjsmin`（pip install rjsmin）
- 封装为 `scripts/minifier.py`
- **涉及文件**：
  - `scripts/minifier.py` — **新增**

### Step 3: 集成到构建流程
- `update_report.py` 在注入数据后、写入 HTML 前，自动压缩内联样式和脚本
- 开发模式（无 `--publish`）默认不压缩，`--minify` 强制压缩
- 发布模式（`--publish`）默认压缩
- **涉及文件**：
  - `scripts/update_report.py` — 修改，构建流程接入压缩
  - `config/config.yaml` — 新增 `minify` 配置节

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 体积缩减 | 对比压缩前后 HTML 大小 | CSS+JS 体积减少 >= 30% |
| 功能无损 | 打开压缩后的 HTML | 样式、K 线图、交互全部正常 |
| 开发模式 | `python update_report.py` | 输出未压缩的 HTML（便于调试） |
| 发布模式 | `python update_report.py --publish` | 输出压缩的 HTML |
| 回归 | 跑一次完整报告 | 报告内容无变化 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 压缩破坏 JS 逻辑 | 低 | 图表不渲染 | rjsmin 是成熟库，不支持代码变换只删空白，很安全 |
| 压缩破坏 CSS | 低 | 样式错乱 | 同上，csscompressor 只做安全优化 |
| 调试困难 | 低 | 压缩后代码不可读 | 开发模式不压缩，发布模式才压缩 |
