# REQ-379: MDD 约束治理 — 风控主角的显式化

**优先级**: 🟡 P1 | **状态**: ✅ 已交付（2026-07-19） | **目标版本**: 待定 | **日期**: 2026-07-17

## 背景

REQ-378 关闭结论确立：牛熊二分器只是风险旋钮，不是收益开关。这意味着**真正的风控主角是 MDD 约束**——bear=1.0、MA18 等档位实际上都是被"MDD ≥ 基线-2pp"这条约束反推出来的。但这条约束本身从未被系统性审视过：

- 前端槽位展示用 [-40, -20]，扫参协议用"基线-2pp"（随基线漂移：-39.4 → -40.4 → 现在约 -40.9），两个口径并存
- 约束值的依据是什么？两融维保比例的硬边界（强平线）？心理承受？还是历史习惯？
- 约束是研究协议里的软约束，不是引擎/产品层的显式参数——它的地位与它的实际权力不匹配

## 讨论点

1. **约束值的第一性依据**：从两融账户维保比例反推（参考 `docs/knowledge/margin-leverage.md` 的 180% 杠杆上限推导），真实的"不可逾越 MDD"是多少？
2. **口径统一**：槽位 [-40,-20]、基线-2pp、心理值——收敛成一个有据可依的数
3. **约束的层级**：留在研究协议（现状）/ 升级为引擎参数（回测中监控、越界告警）/ 进入产品层（实盘断路）
4. **随基线漂移的问题**："基线-2pp"会随着 AR 优化不断放松绝对值（基线 MDD 变差时约束跟着变差），这是不是反了？

## 备注

- 与 REQ-381（reactive 断路器）相邻：约束是"事前预算"，断路器是"事中执行"
- 来源：2026-07-17 二分器研究收口讨论（REQ-378）

## 决策与落地（2026-07-19）

- **约束值**：-40%（绝对值），替代"基线-2pp"
- **直觉背景**：用户美股经历（净值归零）→ 核心关切是"数学上无法翻身"而非心理/维保。修复期 ≈3 年（16% 年化从 -40% 回到水面）。不写进公式，直接写死一个数
- **口径统一**：Tuner 槽位 [-40,-20]、扫参 MDD 约束、gam-0 硬边界——三者在同一个数值上汇合
- **层级**：研究协议层（`research_utils` consume + `quant_universe.yaml` anchor），不进引擎/产品层
- **"基线漂移"消除**：绝对值锚定 → 约束不再随每次采纳的基线 MDD 恶化而自动放松
- **落地**：`config/quant_universe.yaml` 全局 confidence 块 `mdd_cap: -40`；`research_utils._check_constraints` 读此值替代 `baseline_MDD - 2.0`；默认值留 -50 作为兜底（无 YAML 时生效）
- **范围限制**：`optimize_group` 的 `mdd_bound` 参数优先级高于 `mdd_cap`（显式传参覆盖全局），保留调用方灵活性
