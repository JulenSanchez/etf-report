## 动机

当前系统没有任何自动化测试。每次改动（修 bug、加功能、重构）都可能引入回归问题，只能靠人工跑一次报告来验证。随着需求增多，回归风险会越来越大。

**现状问题**：
- 4 个核心脚本没有测试覆盖
- 改一个函数可能影响下游链路，无法提前发现
- 没有测试 = 没有安全网，每次改动都提心吊胆

**不做会怎样**：后续需求（尤其是 REQ-108 多源切换）改动面大，没有测试兜底很容易出 regression。

## 实施步骤

### Step 1: 搭建测试框架
- 引入 pytest（纯标准库 + pytest，无其他依赖）
- 创建 `tests/` 目录结构：
  ```
  tests/
  ├── conftest.py          # 公共 fixture
  ├── test_fix_ma.py       # MA 计算测试
  ├── test_realtime.py     # 实时数据解析测试
  ├── test_config.py       # 配置管理测试
  └── test_update_report.py # 主控流程测试
  ```
- **涉及文件**：
  - `tests/conftest.py` — **新增**，mock 数据和配置 fixture
  - `scripts/../requirements.txt` 或 `pyproject.toml` — 添加 pytest 依赖
- **预期产出**：`python -m pytest` 可运行

### Step 2: fix_ma_and_benchmark.py 测试
- MA5/MA20/MA50 计算正确性（手动算几个值对比）
- 预热逻辑（不足 50 天数据时不崩溃）
- 基准指数数据对齐
- 边界：空数据、单条数据、恰好 50 条数据
- **涉及文件**：
  - `tests/test_fix_ma.py` — **新增**
- **预期产出**：覆盖均线计算核心函数

### Step 3: realtime_data_updater.py 测试
- API 响应解析（正常/异常/空数据）
- 涨跌幅计算精度

- **涉及文件**：

- **预期产出**：覆盖数据解析核心函数

### Step 4: config_manager.py 测试
- YAML 加载、缺省值、环境变量覆盖
- `get_etfs()`、`get_api_config()` 等接口
- **涉及文件**：
  - `tests/test_config.py` — **新增**
- **预期产出**：覆盖配置读取逻辑

### Step 5: update_report.py 测试
- 主流程编排（mock 子模块调用）
- HTML 注入替换是否正确

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 框架可用 | `python -m pytest` | 发现并运行所有测试 |
| 覆盖率 | `python -m pytest --cov=scripts` | 核心模块 >= 80% |
| 全部通过 | `python -m pytest -v` | 100% pass |
| 回归安全 | 跑一次完整报告 | 报告正常生成，无副作用 |
| CI | push 到 GitHub | Actions 自动跑测试 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| mock 数据与真实 API 不一致 | 中 | 测试通过但实际运行失败 | 从真实 API 抓一批数据作为 fixture |
| 内部函数没有显式导出 | 中 | 某些函数难以单独测试 | 必要时小幅重构暴露接口 |
| HTML 注入测试脆弱 | 低 | HTML 结构微调就断测试 | 用正则/BeautifulSoup 做宽松匹配 |
