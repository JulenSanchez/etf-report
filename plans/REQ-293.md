# REQ-293: 回测引擎与 Tuner 包装函数结构性拆分

**创建**: 2026-06-18 | **状态**: backlog | **优先级**: 🟠 High | **目标版本**: v4.1.0

## 背景

v3.8.0 发布后工程审计发现，核心逻辑已通过全量测试，但仍存在结构性重构债：

1. `scripts/quant_backtest.py::run_backtest()` 过长，集中承担配置合并、数据准备、因子预计算、调仓循环、NAV 二次计算、指标统计、debug 输出等职责。
2. `scripts/quant_tuner.py::run_tuner_backtest()` 过长，集中承担参数校验、Tuner 参数契约转换、缓存数据准备、统一回测调用、指标计算、前端 payload 组装等职责。
3. 当前没有功能性阻塞，`python -m pytest tests -q` 已通过；但后续 v4 多主体策略治理、payload 管线、人设仪表盘会继续扩大回测与展示链路，若不拆分，后续改动回归面会变大。

## 目标版本判断

推荐目标版本：**v4.1.0**。

原因：

- 不放入 v4.0.0：v4.0.0 已有 REQ-270/272 等旗舰功能，且当前主链测试全绿；在 v4.0.0 前做大拆分会增加发布风险。
- 不降到 wishlist：该债务位于量化主链，影响后续策略治理、版本演化、Tuner 与正式页 payload 的长期可维护性。
- v4.1.0 合适：等 v4.0.0 的多主体策略形态落定后，再按真实调用边界拆分，能避免提前抽象错方向。

## 目标

1. 在不改变策略结果的前提下拆分 `run_backtest()` 的内部职责。
2. 在不改变 Tuner API 返回结构的前提下拆分 `run_tuner_backtest()` 的内部职责。
3. 让配置解析、数据准备、回测执行、指标汇总、前端 payload 组装各自有清晰边界。
4. 建立结果级回归，证明拆分前后核心 preset 输出一致。

## 非目标

- 不改变因子公式、仓位分配、成交口径。
- 不改变 Tuner UI 交互。
- 不重写量化引擎算法。
- 不引入新策略参数。
- 不做性能优化，除非拆分后自然带来小幅收益。

## 推荐拆分边界

### `scripts/quant_backtest.py`

建议拆为：

```text
_resolve_backtest_config(...)
_prepare_backtest_data(...)
_precompute_backtest_factors(...)
_run_signal_pass(...)
_run_nav_pass(...)
_summarize_backtest(...)
```

保留：

```python
run_backtest(...)
```

作为唯一公共入口，避免影响 CLI、Tuner、payload builder 和测试。

### `scripts/quant_tuner.py`

建议拆为：

```text
_prepare_tuner_preloaded(...)
_call_tuner_backtest_engine(...)
_compute_tuner_metrics(...)
_build_tuner_payload(...)
```

保留：

```python
run_tuner_backtest(params)
```

作为 `/api/run` 的稳定入口。

## 验收标准

1. 拆分前后以下命令核心指标一致或在既有容差内：

```bash
python scripts/quant_backtest.py --preset gam-1 --start 2023-01-01
python scripts/quant_backtest.py --preset act-1 --start 2023-01-01
python scripts/quant_backtest.py --preset zen-1 --start 2023-01-01
```

2. Tuner 契约测试通过：

```bash
python -m pytest tests/test_quant_contract.py -q
```

3. 量化主链测试通过：

```bash
python -m pytest tests/test_quant_* -q
```

4. 全量测试通过：

```bash
python -m pytest tests -q
```

5. `run_backtest()` 与 `run_tuner_backtest()` 保持外部签名不变。

## 风险

- 拆分时容易改变中间状态引用、缓存复用或日期边界，导致结果漂移。
- Tuner 与 CLI 使用不同数据入口，必须同时验证两条路径。
- 如果在 v4.0.0 人设仪表盘前抢做，可能与 v4 payload 需求互相打架。

## 推荐推进方式

1. 先写结果级基线快照，固定 gam-1 / act-1 / zen-1 的关键指标。
2. 先拆纯函数边界，不移动算法细节。
3. 每拆一层跑 `tests/test_quant_*`。
4. 最后只做命名和文档收口，不顺手改策略逻辑。
