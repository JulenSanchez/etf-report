## 现状

### 原始 profile (2026-06-26, 6天回测窗口)

```
总耗时: 50.5s
  _precompute_factors: 40.9s (80%)  ← Python for loop × 54 ETFs × ~1500 rows
  price_lookup iterrows: 6.7s (13%) ← pandas 反模式
  主循环:                2.8s ( 6%) ← 已经很轻
```

### 已完成的优化 (#1 + #2, 2026-06-26)

| # | 优化 | 方法 | 结果 |
|---|------|------|------|
| 1 | 因子缓存 | pickle `data/quant/.factor_cache` | 40s → 0.5s (cache hit) |
| 2 | 去 iterrows | `to_dict('records')` 替代 | 6.7s → 0.7s |

**cache hit 时**: 回测 2-3s，达到目标。
**cache miss 时**: 仍要重算全部因子，53M 函数调用。

### cache miss 路径实测 (2026-06-29, 3Y 窗口, 盘中数据后)

```
总耗时: 63.4s
  _precompute_factors: 35.5s (56%)
    ├─ F7 rolling z-score:  ~17s  ← O(n²) Python loop (12万次迭代)
    ├─ F6 衰竭循环:          ~6s  ← 嵌套 Python loop
    ├─ F1 逐日循环:          ~5s  ← 12万次 Python 迭代
    └─ F3 滚动计算:          ~4s  ← pandas rolling (OK)
  backtest 主循环:         23.5s
    └─ get_price_on_date:   4.4s (8370次 lookup)
  rest:                     4.4s
```

### 为什么上次没发现

因子缓存 (#1) 在数据不变时完全掩盖了 `_precompute_factors` 的 O(n²) 问题。cache hit 时回测 2-3s，所有测试都在 cache 命中状态下跑。但 cache 的 key 是整个 `all_daily` —— 拉盘中数据后所有 54 支 ETF 的 CSV 都变 → cache 全部失效 → 54 支 ETF 全部从头重算因子。

**#3 向量化当时列入但未执行**，因为 cache hit 后的性能已达标（2-3s），cache miss 路径的严重性未被识别。

### Phase 4: 已完成 (2026-06-29) — intraday 缓存稳定性

| # | 优化 | 方法 | 结果 |
|---|------|------|------|
| 6 | **因子/价格数据流分离** | `run_backtest` 新增 `all_daily_exec` 参数。因子计算用 CSV 原始数据（缓存 key 稳定），执行价格查询用 intraday 合并数据 | 缓存不再随 intraday 数据每天失效，命中率从 0% 恢复到 ~100% |

**问题**：`_factor_cache_key` 包含 `daily_df["close"].iloc[-1]`，而 intraday 数据每天更新最后一行 close → cache key 每天变 → 全部 54 支 ETF 重算因子。

**修复**：不改 cache key，而是在上层分离数据流。`run_tuner_backtest` 传给 `_precompute_factors` 的 `all_daily` 保持 CSV 原始数据（稳定），传给 `get_price_on_date` 的 `all_daily_exec` 包含 intraday（执行价格准确）。详见 `quant_backtest.py:477`（`all_daily_exec` 参数）和 `quant_tuner.py:1128`（数据准备）。

**验证** (2026-06-29, 3Y gam-0)：
- Run 1 (cache miss): 40.1s
- Run 2 (cache hit): 25.9s（与 Phase 2 基准 26s 一致）
- 连续多跑 + intraday 数据变动 → 缓存持续命中 ✅
