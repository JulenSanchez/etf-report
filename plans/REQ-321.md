# REQ-321: 帕累托优化引擎落地与 warm-start 机制

**优先级**: 🟠 High | **目标版本**: v4.0.0 | **依赖**: 无 | **创建**: 2026-06-26

## 背景

v3 优化规范已定稿（`docs/runbook/v2-quant/optimization.md`），核心设计：
- 三轮无 prune 收敛（Sobol → TPE → TPE）
- 引擎与预设库解耦
- warm-start 外部传入

但 `scripts/pareto_optimizer.py` 的 warm-start 机制反复失败，已积累的实践经验需要系统消化。

## 已有信息

### 1. warm-start 失败路径（已排除）

| 方法 | 问题 |
|------|------|
| `study.add_trial(FrozenTrial)` | 分布对象不匹配，Optuna 校验拒绝 |
| `prior.enqueue_trial() + prior.optimize()` | 跑完的 trial 参数为空 `{}` |
| `TPE.enqueue_trial()` 后 `study.optimize()` | 种子吃掉 TPE trial 名额 |

### 2. 可行方案（已验证）

| 方法 | 效果 |
|------|------|
| Sobol 缩界 → TPE 从零跑 | ✅ 可行，TPE 在缩小的参数空间内探索 |
| 事后合并 Sobol + TPE 前沿 | ✅ 可行，规避了 trial 注入问题 |
| `--warm-start` 加载 JSON params → 缩界 | ✅ 已验证 |

### 3. 预设库 vs 随机探索

```
gam-0/1/2/3 全部位于 Sobol 50 纯随机无法覆盖的优质区
→ Round 1 必须含 warm-start，纯 Sobol 洒不到
→ 交叉变异（gam-1信号 × gam-2执行）有理论价值但未验证
```

## 目标

1. **`pareto_optimizer.py`** 生产就绪：缩界 warm-start + 无注入 TPE + 前沿合并
2. **预设库接口**：`quant_contract.py` 新增 `presets_as_warm_start(names)` 和 `mark_preset_unreliable(name)`
3. **交叉变异**：实现两两预设参数混合采样
4. **多 preset 共享 Round 1**：一次 Sobol 覆盖 gam-0/1/2/3

## 改动范围

| 文件 | 改动 |
|------|------|
| `scripts/pareto_optimizer.py` | 修复 warm-start（缩界方案）、交叉变异、多 preset 共享 R1 |
| `src/etf_report/core/quant_contract.py` | 新增 `presets_as_warm_start()`、`mark_preset_unreliable()` |

## 验收

```bash
# 1. 缩界 warm-start：50 Sobol + 50 TPE → 产出前沿 > Sobol-only 前沿
python scripts/pareto_optimizer.py --preset gam-2 --warm-start <sobol.json> --r1 0 --r2 50

# 2. 多 preset 共享 R1
python scripts/pareto_optimizer.py --presets gam-0,gam-1,gam-2,gam-3 --r1 80 --r2 50

# 3. 预设库接口
python -c "from etf_report.core.quant_contract import presets_as_warm_start; print(len(presets_as_warm_start(['gam-1','gam-2'])))"
```
