# REQ-120: 基于稳定 ID 的元素指代与轻量调试交互

**优先级**: 🟡 Medium
**估时**: 3-4h
**状态**: done

**目标版本**: -
**依赖**: 已满足（REQ-121）
**创建日期**: 2026-04-13

## 动机

当前协作的核心摩擦不是“页面能不能改”，而是**用户看到的是已注入真实数据的页面，AI 修改的是 HTML 结构**，两边缺少一个**稳定、可复制、可搜索的元素 ID**。

**具体表现**：
- 用户能清楚看到“这个框”“这个卡片”“这个图例”，但很难稳定描述它
- AI 能看懂 HTML 结构，但很难确定用户口中的“这个”到底是哪一个节点
- 页面改动逐渐转向精修间距、节奏、层级、文案位置后，这种沟通损耗会持续放大

因此需要把方案收敛为一件更朴素的事：**尽量为语义稳定、可能成为修改目标的元素补稳定唯一的 `id`，并提供一个轻量交互，让用户能直接复制该 `id` 给 AI。**

## 阻塞说明

本需求此前被阻塞，原因不是交互方案不清楚，而是**单一事实源尚未理顺**。

该阻塞已由 `REQ-121` 在 2026-04-13 解除：
- `update_report.py`、`transaction.py`、`health_check.py`、`verify_html_integrity.py` 已统一围绕根目录 `index.html`
- 技能目录内已清理 `outputs/index.html` 的主流程残留引用

**当前结论**：本需求已具备启动条件，可直接基于根目录 `index.html` 进入实施。

## 目标

在不引入第二套 HTML 源文件的前提下，提供一个面向开发沟通的轻量模式，使用户能低成本、低歧义地告诉 AI：
- 想改的是哪个元素
- 这个元素在源码里的唯一 `id` 是什么
- 这个 `id` 可以直接被 AI 搜索和定位

## 设计原则

- **单一事实源**：继续以根目录 `index.html` 作为唯一页面入口与唯一权威 HTML
- **默认不打扰**：调试功能默认关闭，不破坏正常浏览和交互
- **ID 优先**：核心不是生成路径，也不是发明标签，而是尽量为稳定目标元素提供唯一 `id`
- **正式源码可追溯**：复制给 AI 的内容必须能直接在正式源码里搜索到
- **只做轻量交互**：Phase A 只做高亮、复制 `id`、轻提示，不做复杂 Inspector

## 预期能力

### Phase A：最小可用版
- 在根目录 `index.html` 中为语义稳定的高价值目标元素补稳定唯一 `id`
- 提供轻量调试交互，但默认关闭
- 鼠标悬停高亮当前目标元素
- `Alt+Click` 复制当前目标元素的 `id`
- 复制结果可直接粘贴到 AI 对话中使用

### Phase B：增强版（如仍有需要）
- 可视化开关或更清晰的调试态提示
- 更丰富的复制反馈
- 更细粒度的高亮策略
- 仅在确有必要时再讨论更复杂的辅助面板

## 修复后重开讨论焦点（2026-04-13）

### 1. 运行位置
- 不再要求独立的 `outputs/debug/index.html`
- 调试能力直接基于根目录 `index.html`
- 页面内容不再复制两份，避免额外产物和流程心智负担

### 2. 开关方式
- 调试能力默认关闭
- 优先采用快捷键方式进入/触发调试能力
- `Alt+Click` 作为复制动作，不劫持普通点击
- 正式版可以保留轻量 runtime，但在未进入调试态时不应产生可感知影响

### 3. ID 覆盖策略
- 尽量为**语义稳定、可能成为修改目标**的元素补稳定唯一 `id`
- 第一批优先覆盖：各 `section` 外框、图表容器、legend 容器、K 线区内框、研究报告卡片、关键表格、关键信息块
- 像主标题这类本来就能靠自然语言直接命中的文本，不是第一优先级
- `class` 可以继续用于样式和分组，但**不承担唯一定位职责**
- 不追求给所有细碎叶子节点都加 `id`；优先覆盖真正会被用户点名修改的块

### 4. 复制协议
- `Alt+Click` 默认直接复制目标元素的 `id`
- 不复制路径，不复制额外上下文，不复制调试标签
- 如果当前命中的是文本节点，归一到包裹该文本的最近带 `id` 元素
- 如果当前节点没有 `id`，则向上寻找最近的可修改、且带稳定 `id` 的容器元素
- 因此本需求的核心，不再是“生成一种路径值”，而是“保证想改的元素附近总有合适的稳定 `id` 可被复制”

### 4.1 用户侧表达原则

用户应分两类使用：
- **天然有名字的元素**：继续直接说自然语言本身，例如“持仓股图例”“近期业绩表现”
- **难以描述的框类/容器类元素**：`Alt+Click` 复制对应 `id`，再塞进原句里描述

也就是说：
- **关系由用户自然表达**
- **系统只负责把被点到的对象变成可粘贴的稳定 `id`**
- **AI 再根据自然语言 + `id` 去理解对齐、间距、参照关系**

### 4.2 基于真实提问的理想使用形态

**例 1：前十大持仓股高度对齐**
- 理想用户描述：
  - “修复 bug：`holdings-section-512400` 的高度跟 `fund-overview-section-512400` 不一样高，请让这两个外框对齐；另外理论上对齐了之后，`holdings-legend-512400` 也应该能完整显示，不需要翻页。”

**例 2：研究报告卡片调整**
- 理想用户描述：
  - “把 `report-card-512400-3` 这个卡片的上下节奏调顺一点，它和周围几个卡片看起来不一致。”

### 4.3 鼠标命中与复制目标

- 默认规则：**复制鼠标当前接触到的最近可修改、且带稳定 `id` 的元素**
- 如果命中的是纯文本节点，则归一到包裹它的最近带 `id` 元素
- 如果外框和内框都有各自稳定 `id`，则取当前实际命中的那一层
- 因此 hover 在外框、内框、图表、卡片等不同位置时，复制结果可能不同；这是预期行为

### 4.4 Phase A 的结论

Phase A 最值得优先实现的，就是下面两件事：
- **尽量给语义稳定的目标元素补稳定唯一 `id`**
- **Alt+Click 一个元素时，直接复制它的 `id`**

这会把需求从“解释一套复杂路径体系”降到“给我一个能搜到的唯一 `id`”。

## ID 命名规范（Phase A 初稿）

### 1. 总体规则
- 统一使用 **小写英文 + 连字符** 的 `kebab-case`
- 优先让 `id` 一眼体现 **元素职责**，其次再体现归属范围
- 对 ETF 专属内容，优先把 **ETF 代码** 放进 `id`
- 已经稳定且可搜索的现有 `id` 可以保留，例如 `panel-512400`、`kline-daily-512400`
- 如果现有 `id` 风格明显不统一，应允许顺手收敛，例如把 `holdings-pie-1` 改成更可搜索的 `holdings-chart-512400`

### 2. 推荐结构
- **面板级**：`panel-{scope}`
  - 例：`panel-overview`、`panel-512400`、`panel-macro`
- **ETF 分区级**：`{section}-section-{etfCode}`
  - 例：`fund-overview-section-512400`、`tech-analysis-section-512400`、`holdings-section-512400`、`research-section-512400`
- **区块/组件级**：`{block}-{etfCode}`
  - 例：`performance-table-512400`、`holdings-chart-512400`、`research-rating-512400`
- **重复卡片级**：`{block}-{etfCode}-{index}`
  - 例：`report-card-512400-1`、`report-card-512400-2`
- **概览页卡片级**：`overview-card-{etfCode}`
  - 例：`overview-card-512400`

### 3. 命名优先级
- **第一优先**：语义稳定的业务角色
  - 如 `fund-overview`、`tech-analysis`、`holdings`、`research`
- **第二优先**：稳定归属信息
  - 如 ETF 代码 `512400`
- **第三优先**：只有在同类重复元素无法区分时，才补顺序号
  - 如 `report-card-512400-3`

### 4. 明确避免的命名
- 避免只反映样式的名字：如 `blue-box`、`left-panel`
- 避免脱离业务语义的全局流水号：如 `card-7`、`chart-3`
- 避免把 DOM 位置硬编码进 `id`：如 `left-top-box`
- 避免无意义缩写，除非项目里已经形成稳定习惯

### 5. 结合当前页面的落地建议
- 保留：`panel-512400`、`kline-daily-512400`、`kline-weekly-512400`
- 补齐：`fund-overview-section-512400`、`tech-analysis-section-512400`、`holdings-section-512400`、`research-section-512400`
- 收敛：把 `holdings-pie-1` 这类顺序号命名，改为带 ETF 代码的 `holdings-chart-512400`
- 新增：为研究报告卡片补 `report-card-512400-1..4` 这类稳定 `id`
- 只在真实会被点名修改的容器上补 `id`，不要求给每一层纯装饰节点都命名

## 交互方式



- 调试能力默认关闭，进入页面后不打扰正常浏览
- 当前实现采用**贴底窄底栏**，风格参考常见财经站的底部 ticker
- 底栏从左往右填充多类目占位行情信息，右侧常驻一个可点击元素：`调试功能：开/关`

- 为了不干扰现有 tab 切换、卡片点击和图表交互，复制动作使用 **`Alt+Click`**
- 调试模式开启后，hover 会高亮最近带 `id` 的目标块，并在复制后给出轻提示
- Phase A 先不做右侧 Inspector，优先把“看得见 + 复制得准”跑通



## 正式版隔离策略

- 根目录 `index.html` 仍是唯一发布入口
- 轻量调试 runtime 可以直接与正式页共存，但默认关闭
- 未进入调试态时，不应出现高亮、提示、复制劫持等可感知行为
- 发布前增加一条兼容校验：未进入调试态时，页面交互、样式、图表行为应与当前正式版一致
- 如果后续发现正式页内置 runtime 仍带来负担，再考虑回退到独立调试页；但 Phase A 不再默认走这条路

## 实施步骤

### Step 1: 确认同页调试模式开关
- 基于 `REQ-121` 继续以根目录 `index.html` 为唯一入口
- 确定调试能力的启用方式与默认关闭策略

### Step 2: 补齐稳定 ID
- 优先复用正式源码里已经存在的稳定 `id`
- 对于缺少唯一标识、又确实属于高价值目标的元素，补稳定唯一的 `id`
- 统一采用本需求定义的命名规范：`kebab-case`、优先业务语义、ETF 专属内容尽量带 ETF 代码
- 优先收敛已有不统一命名，例如把 `holdings-pie-1` 这类顺序号命名改为更可搜索的 `holdings-chart-512400`
- `class` 继续承担语义分组与样式职责，不强行把它当作唯一标识
- `id` 命名以**稳定、唯一、可搜索、便于 AI 理解**为准


### Step 3: 注入轻量调试交互
- 注入 hover 高亮样式
- 注入复制能力与提示 UI
- 在 `Alt+Click` 时复制当前目标元素的 `id`
- 保持页面原有 tab、图表、滚动与点击行为不受影响

### Step 4: 定义复制协议
- 默认输出目标元素的原始 `id`
- 不再设计路径值、调试标签或额外结构化载荷
- 仅在未来真实使用中发现 `id` 仍不足时，才重新讨论更复杂的补充协议

### Step 5: 验证沟通效率
- 选取 3-5 个真实页面微调场景做回放
- 验证用户是否能仅靠复制出来的 `id` 完成无歧义指代

## 文件影响分析

| 文件 | 改动类型 | 说明 |
|------|---------|------|
| `index.html` | 修改 | 为高价值目标元素补稳定 `id`，并加入轻量调试交互 |
| `scripts/update_report.py` | 可能修改 | 如有必要，补充调试开关注入或构建控制 |
| `docs/` / `WORKFLOW.md` | 修改 | 补充同页轻量调试模式的工作流说明 |

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| ID 覆盖 | 检查关键容器与卡片 | 想改的高价值目标元素都有稳定 `id` |
| 元素高亮 | 悬停关键元素 | 高亮准确，不误伤布局 |
| 指代复制 | `Alt+Click` 复制 5 个不同元素 | 输出为稳定 `id`，且能在源码中直接搜索到 |
| 正式兼容 | 打开正式页面并保持非调试态 | 页面行为与当前正式版一致 |
| 沟通效率 | 用复制出的 `id` 提 3 次修改需求 | AI 可无歧义定位 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 给过多细碎节点加 `id` | 中 | HTML 噪音增大 | 只优先覆盖高价值、语义稳定、真实会被修改的目标 |
| `id` 命名不稳定 | 中 | 指代失效 | 统一命名规范，优先语义稳定元素 |
| 轻量调试 runtime 干扰正式交互 | 中 | 正式页体验受损 | 默认关闭，并验证非调试态与当前正式版一致 |
| 单靠 `id` 仍不足以覆盖极少数复杂场景 | 低 | 个别需求仍有歧义 | 先跑通最小方案，必要时再补协议，不提前做复杂设计 |
