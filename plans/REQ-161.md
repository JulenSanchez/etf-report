## 背景

2026-04-21 REQ-138 现状核对时，发现三档风险偏好配置卡片（aggressive/moderate/conservative）
虽被 `debug-toolbar.js` L126-131 声明为热区，但 `index.html` 对应内容是完全静态的硬编码，
从未被日更链路写入，造成"热区声明 ≠ 实际实现"的错位（登记为 BUG-017）。

经用户澄清，这组数据**不需要日更**，但**可以作为周更支线**以较低频率更新（每周末自动化脚本一次）。
这暴露了一个更底层的缺失：**调试工具栏当前的热区模型只有"日更/非日更"二元，缺少中间档位**，
导致任何非日更但仍需定期更新的内容都无处安放。

本 REQ 打包两件事：

1. **工具侧**：把火焰按钮重构为"更新节律雷达"，支持同时显示多种更新频率的 selector 分组
2. **业务侧**：把三档 allocation 接入周更支线（独立 yaml + 独立脚本 + 周末自动化）

BUG-017 的 selector 剥离动作（把 6 条 allocation selector 从 `HOT_ID_PATTERNS` 剥出去）
并入子任务 1 一起落地，保持 BUG-017 判词层面闭环 + 代码层面由本 REQ 清理。

## 设计：方案 B —— 单键多色"更新节律雷达"

### 视觉与交互

- 火焰按钮 🔥 语义从"显热数据"扩展为"显更新节律雷达"
- **单键切换**：点一下进入"节律显形"，再点一下退出
- **一次性同时显示多档**，用不同高亮色区分：

| 档位 | 代表色 | 语义 | 示例 id |
|---|---|---|---|
| 🔴 日更 daily | 粉色 `#ff4d9f`（沿用现状） | 每次 `update_report.py` 会改 | `performance-return-*`、`report-card-content-*` |
| 🟡 周更 weekly | 青色 `#4dc7ff` | 每周末独立脚本刷新 | `*-allocation-card-item-*`、`*-allocation-card-strategy` |
| 🟢 月更 monthly | 绿色 `#52c41a`（预留） | 更低频，当前无实例 | — |
| ⚪ 静态 static | 不框 | 从不变 | 页面骨架、标题文案 |

- 三色同时在屏上出现 → 用户一眼看清"哪些今天变、哪些周末变、哪些从不变"
- tooltip / 悬停文案需补颜色图例（类似 REQ-145 给热区补图例的做法）

### 取舍理由（为何选 B 而非新增独立按钮）

- 新增 🗓️ 周更按钮 = 工具栏再占一格，当前三图标刚按 REQ-142/143/144 收拢，不想回退
- 多档共显让用户有"更新节律全景"的直觉，比分多个按钮反复切换体验更连贯
- 语义统一性：火焰 = 更新节律，颜色细分 = 频率档位，一层嵌套关系清晰

## 子任务拆分

### 子任务 1：调试工具栏节律雷达重构（纯前端 / 可 Claude 侧评审 + CB 侧落地）

**核心改动：`assets/js/debug-toolbar.js`**

1. 把现有 `HOT_ID_PATTERNS` 按节律拆分：
   ```js
   const DAILY_ID_PATTERNS = [ /* 原 HOT_ID_PATTERNS 去掉 6 条 allocation */ ];
   const WEEKLY_ID_PATTERNS = [
       { selector: '[id^="aggressive-allocation-card-item-"]', allowChildren: true },
       { selector: '[id^="moderate-allocation-card-item-"]', allowChildren: true },
       { selector: '[id^="conservative-allocation-card-item-"]', allowChildren: true },
       '#aggressive-allocation-card-strategy',
       '#moderate-allocation-card-strategy',
       '#conservative-allocation-card-strategy',
   ];
   const MONTHLY_ID_PATTERNS = [];  // 预留
   ```

2. `highlightHotspots()`（或等价入口）扩展为三组独立着色：
   - 日更组：沿用粉色 outline
   - 周更组：改用青色 outline
   - 月更组：绿色 outline（当前空，占位即可）

3. tooltip 文案更新（参考 REQ-145 给热区 tooltip 补颜色含义的做法），
   说明"粉=日更 / 青=周更 / 绿=月更"。

4. **BUG-017 selector 剥离**：L126-131 的 6 条从 `HOT_ID_PATTERNS` 移除，
   落位到 `WEEKLY_ID_PATTERNS`。这一步即是 BUG-017 代码层面的收尾。

5. 回归测试（若有 js 测试套件）：确保节律雷达开关、高亮渲染、tooltip 图例正常。

**额外兼容点**：
- `debug.css` 若有 `.hotspot-highlight` 类的粉色硬编码，需抽出三组类（`.hotspot-daily` / `.hotspot-weekly` / `.hotspot-monthly`）
- 状态持久化（localStorage）按现有 `HOTSPOT_STORAGE_KEY` 方式处理，不改 key 名（向前兼容）

### 子任务 2：三档配置周更支线接入（CB 侧专属 / 涉及自动化调度）

**必改文件**：

1. **新增 `config/allocation_content.yaml`**
   - 三档 × 3 条 item + 1 条 strategy
   - 顶层 `content_date: 'YYYY-MM-DD'` + `freshness_policy: weekly`
   - 首版初值直接照搬 `index.html` L671-690 当前静态文案，保持页面显示不变

2. **新增 `scripts/update_allocation_weekly.py`**
   - 独立可执行脚本（`python scripts/update_allocation_weekly.py`）
   - 读 `allocation_content.yaml`，用 `update_report.py` 里既有的 `_replace_element_by_id`
     把三档 item/strategy inner HTML 写入 `index.html`
   - 复用 `_evaluate_editorial_freshness`（L834-882 已有 `weekly` 分支），
     `content_date` 距今 > 7 天时 strategy 末尾追加"· 周更观点待复核"提示
   - 与 `update_report.py` **不共享**主流程——各自独立跑，互不依赖
   - `logger.info` 输出 yaml 加载与三档写入结果

3. **`scripts/update_report.py`** 无改动（日更链路不碰三档数据）

4. **`.codebuddy/automations/` 新增周末定时任务**
   - 每周六（或周日）上午跑一次 `update_allocation_weekly.py`
   - 完成后按 etf-report 正常发布通道推送（与日更通道独立，避免污染日更节奏）
   - 若 yaml 未改动：脚本应幂等（写出内容与当前 HTML 一致 → git 无 diff → 不触发发布），
     避免空推送
   - 若 yaml 有改动：走常规 `git commit/push` → GitHub Pages 自动部署

5. **`verify_html_integrity.py` / `health_check.py`**（可选）
   - 允许识别 `freshness_policy: weekly` 的 id，不将"未按当天 `data_cutoff_date` 对齐"判为 FAIL
   - 当前这两个脚本似乎并未校验三档 allocation id，可能本子任务无需触碰它们
