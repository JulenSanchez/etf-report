**依赖**: REQ-121, REQ-120
**创建日期**: 2026-04-15

## 动机

当前 `index.html` 已接入 `klineData` 与 `etf_realtime_data.json` 的部分生成流程，但用户可见的很多关键数字和结论仍直接硬编码在 HTML/内联脚本里。

这导致一个明显问题：**数据文件每天在更新，但报告页展示的很多核心信息并没有跟着更新**。典型例子包括：

- 总览卡 `overview-card-512070` 仍显示 `-6.34%`
- 详情页 `512070` 的 `日涨跌幅` 仍显示 `+1.93%`，而实时数据已是 `-0.5%`
- 资金流向模块仍写着 `2026-03-17 14:22:40`
- 总览概览图、研究摘要中的收益率数字、推荐/观望结论等仍停留在旧值或静态文案

### 现状问题

- 页面中的“日报级数值”与 `data/` 下的最新数据源脱节
- 主流程文档写着“实时行情 + 日期会更新”，但实际接入只覆盖了 `klineData` 和部分日期字段
- `realtimeData` 在 HTML 中没有成为稳定事实源，`realtime_data_updater.py` 中部分更新函数也未纳入主流程
- 总览图表、技术面评分、研究卡片、资金流向等模块缺少统一的数据口径与更新边界

### 不做会怎样

- 用户会持续看到“报告日期是今天，但关键数字还是旧的”这种明显不可信状态
- 每日更新流程的感知价值下降，页面可信度被削弱
- 后续再接更多数据源时，静态文本与动态数值会继续分裂，维护成本越来越高

## 本轮目标

- 明确哪些页面元素必须随每日更新自动刷新
- 建立“动态数值 / 派生分析 / 编辑态文案”的三层口径
- 将已有数据源能覆盖的页面元素优先接成自动更新
- 为暂时无法自动化的数据块建立结构化配置入口，而不是继续散落硬编码在 HTML 中
- 修正实时数据链路中的已知缺口，避免页面与 JSON 数据继续脱节

## 范围边界

### 本轮纳入

- 总览卡收益率、评级标签、推荐标签
- ETF 详情页基金概览中的最新价格/净值口径、日涨跌幅、近期业绩表
- 概览页图表：近期涨跌幅对比、基金规模对比、近1年收益率排名、评分雷达图
- 持仓饼图中的成分股当日涨跌与“其他”占比
- 技术面状态、评分、相对基准结论等从 K 线派生的描述
- 资金流向模块的时间戳、统计卡、TOP 表、图表数据接线或降级处理
- 研究摘要 / 推荐结论 / 宏观文案的结构化入口设计

### 本轮不纳入

- 引入新的外部行情源
- WebSocket 盘中实时刷新
- 多页面拆分或前后端重构
- 自动生成完整投研文案的大模型方案

## 审查结论摘要

### A. 已有动态源、但页面未接入

- `日涨跌幅`
- `持仓股当日涨跌`
- `近1月/近3月/近6月/近1年收益率`
- 概览页收益率图表与收益率排名图
- 基于 K 线可推导的技术面状态与评分

### B. 页面硬编码且应至少按日报结构化维护

- 总览卡星级、推荐标签
- 行业研究报告卡片中的结论文本和嵌入数字
- 基本面评分 / 综合建议
- 宏观环境、行业配置建议、风险提示

### C. 已发现的链路问题

- HTML 中没有稳定的 `const realtimeData = ...` 数据块
- `update_report.py` 会读取实时数据文件，但未真正把多数实时字段接到页面展示
- `realtime_data_updater.py` 中 `update_html_etf_change()` / `update_html_holdings_pie()` 未纳入主流程
- `realtime_data_updater.py` 构造 `ETF_CONFIG` 时未把 `total_ratio` 带入，导致 `etf_realtime_data.json` 当前输出为 `0`
- 资金流向 `fund_flow_data.json` 仍是预留态，但页面展示成了“像是真的在更新”

## 实施步骤

### Step 1: 建立页面元素动态口径清单
- 按模块梳理 `index.html` 中所有需要日更的元素
- 标注其数据来源：`klineData` / `etf_realtime_data.json` / 新增结构化配置 / 暂无数据源
- 明确哪些字段是“自动算”，哪些字段是“人工维护但需结构化”

### Step 2: 接通纯数值型动态字段
- 将总览卡收益率、近期业绩表、概览页收益率图、年收益排名图统一改为从 `klineData` 派生
- 将 ETF 详情页 `日涨跌幅` 与持仓股当日涨跌统一改为从实时数据注入
- 明确“最新净值/最新价格”的显示口径，避免页面文本误导

### Step 3: 修复实时数据链路缺口
- 把 `realtime_data_updater.py` 中真实有用的 HTML 更新能力并入主流程，或重写为统一注入流程
- 修复 `total_ratio` 丢失问题
- 清理 `outputs/js/main.js` 等遗留分支，避免继续误导主流程设计

### Step 4: 将派生分析改为自动生成
- 从日线/周线数据派生技术面状态、相对强弱、量价关系、评分等结论
- 概览页评分雷达图改为由同一套评分结果驱动，避免图和文字口径分裂

### Step 5: 将编辑态内容结构化
- 为研究摘要、推荐结论、宏观环境、行业配置建议、风险提示建立结构化输入文件
- 页面改为读取结构化内容，而不是把文本散落在 `index.html`
- 若某块内容暂不能日更，需明确标记“数据日期/观点日期”

### Step 6: 资金流向模块做接线或降级
- 若可接入 `fund_flow_data.json`，则把时间戳、统计卡、TOP 表与图表统一动态化
- 若当前不准备维护该数据源，则页面改为明确降级，不再展示伪实时旧值

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 页面上仍残留零散硬编码 | 高 | 用户继续看到旧值 | 先建立模块级清单，再逐块替换，不靠人工记忆 |
| 动态计算口径不一致 | 中 | 图表与表格出现互相打架 | 统一由单一派生函数输出收益率/评分结果 |
| 实时数据链路改造影响现有 HTML 稳定性 | 中 | 报告生成异常 | 保留事务备份 + 增补单测 + 跑完整性检查 |
| 研究文案短期难以全自动化 | 高 | 需求范围失控 | 将其降级为“结构化维护”，先解决可计算字段 |
| 资金流向源不稳定 | 中 | 页面出现空白或旧数据 | 设计显式降级模式，不再伪装成实时数据 |
