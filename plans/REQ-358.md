# REQ-358: 份额变动后的因子缓存自动重建

**优先级**: 🟡 P1 | **状态**: backlog | **目标版本**: v3.12.0

## 背景

REQ-354 已实现拆股自动检测 + 内存清洗，但清洗在内存完成不改变 CSV，导致因子缓存 key 不变、读到旧缓存、回测因子值异常（BUG-042）。

## 链路

```
份额变动事件 (AKShare fund_cf_em)
  → _ensure_splits_detected() 检测
  → _apply_split_memory_bridge() 内存清洗（当前只调内存，不碰 CSV）
  → _full_refetch_split_etfs() 盘后全量重拉（覆盖 CSV）
  → 因子缓存应在重拉后自动失效（末笔价变化 → key 变化）
```

理论上链路最后一环已闭环——盘后重拉 CSV 后数据指纹变、缓存自动失效。但 BUG-042 暴露了中间状态的问题：

1. 拆股当天盘中：内存已清洗但 CSV 未变
2. 盘中跑回测 → 因子缓存命中（key 未变）→ 读到旧缓存 → F1/F7 异常

## 需求

### 1. 清洗后立即清理因子缓存

`_apply_split_memory_bridge()` 执行后，删除受影响的 ETF 的所有因子缓存文件。

### 2. 盘后重拉后确认缓存重建

`_full_refetch_split_etfs()` 完成后，CSV 已更新，缓存已自然失效。无需额外操作，但应记录日志确认。

### 3. 数据管理面板显示拆股标记

因子缓存视图新增"拆股影响中"状态，提醒用户该 ETF 的因子缓存可能不准确。

## 修改文件

| 文件 | 改动 |
|------|------|
| `scripts/quant_tuner.py` | `_apply_split_memory_bridge` 后清理因子缓存；`_full_refetch_split_etfs` 后记录日志 |

## 验证

1. 模拟拆股事件 → 运行回测 → 确认因子缓存被清除 → 因子值正确
2. 数据管理面板因子视图 → 份额变动的 ETF 显示"拆股影响中"
