## 动机

当前系统**唯一数据源是新浪财经 API**。如果新浪接口调整、限流、或宕机，报告完全无法更新。这是一个单点故障（SPOF），需要引入备用源消除风险。

**现状问题**：
- `update_report.py` 硬编码调用新浪 K 线接口
- `realtime_data_updater.py` 硬编码调用新浪实时行情接口
- 源挂了没有任何降级手段，直接报错退出

**不做会怎样**：新浪 API 任何波动都会导致报告中断，且无法快速切换。

## 实施步骤

### Step 1: 调研腾讯财经 API
- 确认可用的 K 线历史接口和实时行情接口
- 整理字段映射关系（新浪字段 → 腾讯字段）
- 确认频率限制和稳定性
- **涉及文件**：无代码改动，输出调研结论
- **预期产出**：一份接口对照表

### Step 2: 抽象数据源接口
- `config.yaml` 新增 `data_sources` 配置节：
  ```yaml
  data_sources:
    primary: sina
    fallback: tencent
    timeout:
      primary: 10
      fallback: 15
  ```
- 抽取 `DataSource` 基类（或协议），定义统一接口：`fetch_kline()`、`fetch_realtime()`
- **涉及文件**：
  - `config/config.yaml` — 新增配置节
  - `scripts/data_source.py` — **新增**，基类 + 工厂函数
- **预期产出**：可配置的多源框架骨架

### Step 3: 实现腾讯 adapter
- 新建 `scripts/tencent_adapter.py`，实现 K 线获取 + 实时行情获取
- 字段对齐到现有 JSON 结构（`etf_full_kline_data.json`、`etf_realtime_data.json`）
- **涉及文件**：
  - `scripts/tencent_adapter.py` — **新增**
- **预期产出**：可独立测试的腾讯数据源

### Step 4: 实现智能源选择 + 故障转移
- 主流程先尝试主源，超时/报错自动 fallback 到备用源
- 日志记录选中源、切换原因、各源响应时间
- **涉及文件**：
  - `scripts/update_report.py` — `run_kline_update()` 改用多源调度
  - `scripts/realtime_data_updater.py` — `fetch_realtime_quote_sina()` 改为多源
- **预期产出**：主源挂了自动切备用，报告仍正常生成

### Step 5: 健康检查适配
- `health_check.py` 的数据源相关检查项适配多源
- 检查结果中展示各源可用性
- **涉及文件**：
  - `scripts/health_check.py` — 修改数据源检查逻辑

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 主源正常 | 正常跑 `python update_report.py` | 使用新浪源，日志记录 `source: sina` |
| 主源故障 | 模拟新浪超时（改 config timeout=0.001） | 自动切腾讯，报告正常生成 |
| 数据一致性 | 对比新浪/腾讯同一天的 K 线数据 | 收盘价误差 < 0.1% |
| 回归 | 完整跑一次报告 | HTML 报告内容无变化 |
| 健康检查 | `python health_check.py` | 展示各源状态，通过率 >= 88% |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 腾讯 API 字段与新浪不一致 | 高 | 数据解析出错 | Step 1 充分调研，写对照表 |
| 腾讯 API 突然改接口 | 中 | adapter 失效 | adapter 内做好异常处理和日志 |
| 多源数据微小差异 | 中 | 用户困惑 | 日志中明确记录使用了哪个源 |
| 改动范围大 | 中 | 引入回归 bug | 先做 REQ-113 单测，用测试兜底 |
