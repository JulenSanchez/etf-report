# REQ-360: 拆股后数据修复工作流

**优先级**: 🔴 P0 | **状态**: done | **目标版本**: v3.12.0 | **完成日期**: 2026-07-07

---

## 一、背景

### 问题

ETF 份额拆分（如 515880 通信 1:2）后，前复权（qfq）数据在 Tencent API 有传播延迟：

1. 拆股当天，API 返回未调整的历史价格（拆前价）
2. 拆股后的实时价是拆后价
3. 同一支 ETF 的 CSV 中出现价格跳变（如 close 从 1.58 突然变成 0.76）
4. API 可能需要数小时甚至次日才返回调整后的全量数据

### 现状

- REQ-354 已实现：AKShare 拆股检测 + 内存清洗（临时保护）+ 盘后全量重拉
- 缺失：盘后重拉时 API 可能仍未更新 → 拉回的数据依然是未调整的 → 用户只能手动修复（如 2026-07-06 通信 ETF）
- BUG-042：内存清洗不改 CSV，因子缓存 key 不变 → 读到旧缓存 → 回测因子异常

### 用户旅程（目标）

```
用户打开 Tuner，点"刷新数据"
  → 系统检测到 515880 7/3→7/6 close 跳变 -52%（异常）
  → 查询 AKShare → 确认 7/3 为 1:2 拆股 ex-date
  → DM 面板标记 515880 为"拆股待修复"
  → 用户看到标记，点击"全量重拉"
  → 系统删除 515880 CSV → Tencent API 全量拉取 → 写入新 CSV
  → 验证 close 序列连续 → 清除"拆股标记"
  → 重建因子缓存
  → 回测正常运行，结果仿佛没发生过份额变动
```

## 二、设计

### 新增后端 API

**`GET /api/split_status`** — 返回各 ETF 的拆股状态

```json
{
  "515880": {
    "has_split": true,
    "ex_date": "2026-07-03",
    "ratio": 2.0,
    "status": "pending_repair",  // pending_repair | repaired | none
    "symptom": "close jump -52%: 1.579→0.757"
  }
}
```

**`POST /api/data_full_refetch`** — 全量重拉（full=True）

```json
{"codes": ["515880"], "verify_split": true}
```
- 删除 CSV → `update_single(full=True)` → 验证 close 连续性 → 清除拆股标记 → 重建因子缓存

### 拆股检测增强

在 `refresh_data()` 和 `data_refetch()` 的 post-market 路径中，追加**异常检测**：

1. 遍历每支 ETF 的最近 N 天 close 序列
2. 检测单日跳变 > 30%（潜在拆股）
3. 命中 → 查 `_SPLIT_EVENTS` 确认 → 标记 `pending_repair`
4. 未命中 split_events → 标记为"异常波动"（非拆股，可能是数据错误）

### 前端

- DM 面板 ETF 行显示拆股图标（⚠）+ tooltip "7/3 1:2拆股, 待全量重拉"
- 选中拆股 ETF 时，操作按钮区显示"全量重拉(拆股修复)"按钮
- 全量重拉完成后 toast 通知"515880 拆股数据已修复"

## 四、验证方案（假数据注入）

### 场景模拟

通信 ETF 515880 真实发生 1:2 拆股（ex-date 2026-07-03）。模拟拆股前未复权的状态：

### Step 1: 构造假数据

```
1. 读取 515880_daily.csv
2. 将 2026-07-03 之前所有行的 close/open/high/low × 2（模拟拆前价）
3. 删除 2026-07-06 和 2026-07-07 的行（模拟拆股后数据未拉取）
4. 写回 CSV
5. 删除 515880 的因子缓存文件
```

### Step 2: 触发异常检测

```
1. 点"刷新数据"（或数据管理面板"补全空缺"选中 515880 的 7/6-7/7）
2. 预期：系统检测到 7/3 close=1.58 → 7/6 close=0.76（-52%）
3. 预期：查询 AKShare → 确认 1:2 拆股
4. 预期：DM 面板 515880 行显示 ⚠ "拆股待修复"
```

### Step 3: 全量重拉修复

```
1. 选中 515880 → 点"全量重拉(拆股修复)"
2. 预期：delete CSV → Tencent API full=True 拉取 → 写入新 CSV
3. 预期：验证 close 序列连续（7/3 close 和 7/6 close 在合理范围）
4. 预期：自动清除"拆股标记"
5. 预期：重建因子缓存
```

### Step 4: 回测验证

```
1. 运行 gam-0 回测，end_date=2026-07-06
2. 预期：回测正常运行，无 NaN 异常
3. 预期：515880 的 F1/F7 值在合理范围（非 BUG-042 的 F1=93.97）
4. 预期：最终 NAV 与正常数据一致（差 < 1%）
```

### Step 5: 恢复数据

```
1. git checkout data/quant/515880_daily.csv  # 恢复原始文件
2. 删除因子缓存 → Tuner 重启自动重建
```
