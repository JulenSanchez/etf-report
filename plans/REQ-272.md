# REQ-272: 版本演化协议（软分叉）

**优先级**: 🟠 P1
**估时**: 6-8h
**状态**: backlog
**目标版本**: v4.0.0
**依赖**: REQ-271（并行化，三派独立评估需要并行回测）、REQ-270（仪表盘，提案和投票状态需要在 dashboard 展示）
**创建日期**: 2026-06-03
**最后活动**: 2026-06-03

---

## 动机

> v4 的"多主体"不能只是三组静态参数——它们需要有能力对变化做出反应。当新因子被提出、现有因子被质疑、或市场结构发生变化时，三个人设应该各自表态。这是"策略治理"从人工走向半自动的第一步。

**现状问题**：
- 三个人设的参数是手动锁定的，没有升级/降级机制
- 新因子（如 F7）引入时，没有系统化的"谁接受、谁拒绝、为什么"的记录
- 策略版本的演化历史分散在 research/ 文档中，没有结构化

**不做会怎样**：
- 每次策略调整都是一次性的"手动改 YAML"，没有可追溯的决策链
- 三个人设永远停留在 2026-05-28 的参数上，随时间推移必然过时

## 核心设计约束（2026-06-03 用户确认）

1. **不允许真分叉**：引擎逻辑统一，只在参数层面分叉
2. **新因子默认进入引擎**：不接受的人设设权重=0
3. **分歧由用户/中心 AI 裁决**：会话内手动决定
4. **策略版本记录**：谁在用哪个版本的哪组参数

## 实施步骤

### Step 1: 策略版本结构化存储
- 新建 `config/strategy_versions.yaml` 或复用 `quant_universe.yaml`
- 记录每个 preset 当前使用的版本号和参数快照
- 版本历史链：`version: 1 → 2 → 3`，每次参数变更新增一条记录
- 涉及文件：`config/strategy_versions.yaml`（新建）

### Step 2: 提案数据结构
- 定义提案格式（YAML/JSON）
- 提案类型：新增因子、修改因子映射、调整执行逻辑、改置信函数
- 每个提案包含：提案人、日期、变更内容、影响评估
- 涉及文件：`config/proposals/`（新建目录）

### Step 3: 三派投票逻辑
- 提案触发 → 自动跑三派回测（用并行化基础设施）
- 各派用自己的目标函数评估：
  - 精算师：提案是否提升等权 S×C？
  - 禅修者：提案是否提升等权 Sharpe？
  - 赌徒：提案是否提升等权 AR？
- 投票结果：accept / reject / abstain
- 涉及文件：`scripts/quant_proposal.py`（新建）

### Step 4: 仪表盘集成
- REQ-270 仪表盘的"§5 演化立场"消费提案和投票数据
- dashboard payload 包含当前活跃提案 + 各派投票状态
- 涉及文件：`scripts/build_dashboard_payload.py`

### Step 5: 用户裁决入口
- 当三派出现分歧（有人 accept 有人 reject），仪表盘展示分歧状态
- 用户在 Claude Code 会话内手动裁决
- AI 根据裁决结果更新 `strategy_versions.yaml`

## 验证方式

| 验证项 | 方法 | 通过标准 |
|--------|------|---------|
| 提案评估 | 提交一个已知效果的参数变更 | 三派投票结果与预期一致 |
| 版本记录 | 应用一次变更后查看 strategy_versions.yaml | 新增版本记录，包含旧参数快照 |
| 仪表盘展示 | 有活跃提案时打开 dashboard | §5 正确显示提案和投票状态 |
| 回退能力 | 手动回退到旧版本参数 | 回测结果与历史记录一致 |

## 风险与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 提案评估耗时过长 | 高 | 用户等待不耐烦 | 依赖 REQ-271 并行化，三派同时评估 |
| 参数版本膨胀 | 低 | 配置文件过大 | 只保留最近 N 个版本，旧版本归档 |
| 投票结果与直觉不符 | 中 | 用户对系统失去信任 | 投票附带详细理由（目标函数变化量），用户始终有否决权 |
