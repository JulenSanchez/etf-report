## 🎯 核心设计原则

### 1. 数据+模板分离

**原则**: 数据与展示层完全分离

**实施**:
```
HTML 模板（样式源）
    ↓
JavaScript 对象容器
    ↓
数据注入（JSON）
    ↓
运行时渲染
```

**优点**:
- ✅ 样式稳定，不因数据更新而变化
- ✅ 数据更新独立于 HTML 修改
- ✅ 支持多种数据源无需改 HTML

**示例**:
```javascript
// HTML 中的容器
<script>
  const klineData = {...};  // 注入的数据
  const realtimeData = {...};  // 运行时填充
</script>
```

---

### 2. 配置化管理

**原则**: 所有硬编码参数提取到配置文件

**实施**:
- `config/config.yaml` - 主配置（330+ 行）
- `config/holdings.yaml` - 成分股配置（350+ 行）
- 环境变量覆盖支持

**参数覆盖**:
```
环境变量（最高优先级）
    ↓
命令行参数
    ↓
config.yaml
    ↓
代码默认值（最低优先级）
```

**优点**:
- ✅ 新增 ETF: 4 小时 → 15 分钟
- ✅ 支持多环境部署
- ✅ 无需修改代码

**配置层级**:
```yaml
# config.yaml 结构
etf_codes:
  - code: sh512400
    name: 有色金属ETF
    market: sh
    benchmark: sh000300

timings:
  k_line_fetch_delay: 0.5
  realtime_fetch_delay: 0.3

output:
  html_path: deploy/index.html
  log_path: logs/
```

---

### 3. 结构化日志

**原则**: 所有日志采用统一格式，支持机器解析

**实施**:
- JSON Lines 格式（JSONL）
- 多级别日志（DEBUG/INFO/WARN/ERROR）
- 彩色终端输出 + 文件输出

**日志结构**:
```json
{
  "timestamp": "2026-04-07T16:44:30.123",
  "level": "INFO",
  "source": "update_report",
  "message": "K线数据获取完成",
  "data": {"etf_count": 6, "duration": 3.2},
  "stack_trace": null
}
```

**优点**:
- ✅ 机器可解析，便于分析
- ✅ 结构化存储，便于搜索
- ✅ 完整的执行链路追踪

---

### 4. 系统健康检查

**原则**: 自动检测系统状态，及早发现问题

**实施**:
- 25 项自动检查
- 覆盖文件、数据、依赖、HTML、工作流、配置
- 彩色表格输出 + JSON 报告

**检查分类**:

| 类别 | 检查项数 | 意义 |
|------|---------|------|
| 文件完整性 | 5 | 必要文件是否存在 |
| 数据有效性 | 5-6 | 数据是否完整和正确 |
| 脚本依赖 | 4-5 | 必要库是否安装 |
| HTML 结构 | 3-4 | HTML 是否合法 |
| 工作流逻辑 | 3 | 执行链路是否正确 |
| 系统配置 | 2 | 配置是否有效 |

**优点**:
- ✅ 88% 通过率正常
- ✅ 失败时给出具体原因
- ✅ 自动集成到执行流程

## 📊 数据流设计

### Step 1: 数据输入

```
新浪财经 API
    ↓
K线数据（原始）
    ↓
实时行情数据（原始）
```

### Step 2: 数据处理

```
K线数据 + 实时数据
    ↓
均线计算（MA5/MA20/MA50）
    ↓
数据验证与清洗
    ↓
基准指数对标
    ↓
处理结果（JSON）
```

### Step 3: 数据存储

```
JSON 数据
    ↓
etf_full_kline_data.json（K线 + 均线）
etf_realtime_data.json（实时行情）
```

### Step 4: 数据输出

```
JSON 数据
    ↓
HTML 注入
    ↓
index.html（最终报告）
    ↓
发布（企微 + GitHub）
```

## 📋 设计决策记录 (ADR)

### ADR-1: 为什么采用数据+模板分离？

**背景**: HTML 样式复杂，难以维护

**选项**:
1. ❌ 每次都重写整个 HTML
2. ❌ 使用模板引擎（Jinja2 等）
3. ✅ JavaScript 对象注入

**决策**: 采用选项 3

**理由**:
- 最小化改动
- 100% 保证样式
- 无额外依赖

---

### ADR-2: 为什么提取配置到 YAML？

**背景**: 115+ 硬编码参数散落在代码中

**选项**:
1. ❌ 保持现状
2. ❌ 使用 Python ConfigParser
3. ✅ YAML 配置文件

**决策**: 采用选项 3

**理由**:
- 新增 ETF: 4 小时 → 15 分钟
- 支持环境变量覆盖
- 易于版本控制

---

### ADR-3: 为什么使用结构化日志？

**背景**: print() 满天飞，难以分析

**选项**:
1. ❌ 保持 print()
2. ❌ 使用 logging 模块（标准日志）
3. ✅ JSON Lines 格式

**决策**: 采用选项 3

**理由**:
- 机器可解析
- 支持时间戳和上下文
- 便于集成监控

---

### ADR-4: 为什么有 25 项健康检查？

**背景**: 执行失败原因难以定位

**选项**:
1. ❌ 无检查
2. ❌ 简单的异常捕获
3. ✅ 全面的系统检查

**决策**: 采用选项 3

**理由**:
- 问题早期发现
- 自动化诊断
- 88% 通过率正常
