# 反思区

记录 AI 在协作中反复出现的故障模式，与 etf-report 本体开发无关。

## 镇 AI 法器

**`GPT协作评审.md`** —— 第三方审计视角的 AI 行为评估与工作协议。

- 定位：长期维护的**核心约束文档**。当 AI 进入辩论循环、抵抗用户指令、或用实现细节混淆业务语义时，用户只需说"读 GPT 规范"，AI 应停止当前行为并逐条自检。
- 维护方式：每次发现新的协作故障模式，应在 §2 中新增或扩展现有条目，在版本记录中标注日期和变更摘要。案例文件（见下方）中的典型现象可提炼为通用模式补充到 §2。
- 触发词：`读 GPT 规范`、`看 GPT 评审`、`停下来读协作规范`。

## 案例库

案例文件用于**记录具体翻车现场**，为 `GPT协作评审.md` 的通用模式提供事实证据。案例文件名格式：`YYYY-MM-DD-简短描述.md`。

### 案例格式

```
# YYYY-MM-DD: 一句话标题

## 现象
用户看到了什么

## AI 的错误
具体的错误行为

## 根因
为什么反复犯错

## 正确做法
应该怎么做

## 适用信号
如何判断自己处于类似状态
```

### 新增案例的触发条件

- 同一行代码/同一个问题被修正超过 3 轮
- 用户明确指出"你理解错了"且事后证明用户是对的
- AI 和用户在**不同语义层**交谈（AI 讲实现，用户讲业务）
- 出现了 `GPT协作评审.md` 未覆盖的新故障模式

### 案例与 GPT 协作评审的关系

案例 → 提炼通用模式 → 补充到 `GPT协作评审.md` §2。案例是"这一次发生了什么"，评审是"这一类问题是什么模式"。评审是活的——每次值得记录的翻车都应该让它更完整。
