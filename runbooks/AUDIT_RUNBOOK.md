# 项目审计规程

**版本**: 4.1 | **最后更新**: 2026-06-02

> 本文是 etf-report 项目的审计方法论。它描述**如何审计**，不描述**审计什么**。
> 具体文件名、函数名、API 端点均属于项目自身——AI 应在审计时自行发现它们，而非从本文中读取。
> 本文只有项目结构发生范式级变化时才需要修订。

---

## 0. 设计原则

本文的稳定性来自一条约束：**只写方法，不写目标**。

| ❌ 不稳定（会过时） | ✅ 稳定（长期不变） |
|---|---|
| "审计 `quant_backtest.py` 的 `_execute_rebalance()`" | "审计回测引擎中的核心调仓函数" |
| "检查 `/api/run` 端点" | "追踪一条用户-facing API 从入口到输出的完整链路" |
| "grep `F2`/`F4`/`F5`" | "从项目配置中读取退役对象清单，然后逐一 grep" |
| "读取 `quant_universe.yaml`" | "读取当前活跃的配置文件，建立基线" |

---

## 1. 审计架构

审计从四层推进。**L0 是主体**，L1-L3 是补充。

| 层 | 名称 | 权重 | 回答的问题 |
|----|------|:---:|-----------|
| **L0** | 代码审计 | **40%** | 逻辑有没有 bug？有没有冗余？架构干净吗？前端好用吗？ |
| **L1** | 边界审计 | 15% | 有没有不该出现的东西？ |
| **L2** | 一致性审计 | 20% | 文档说的和代码做的一样吗？ |
| **L3** | 质量审计 | 25% | 代码/参数/文档/前端在工程上好吗？ |

执行顺序：L0 → L1 → L2 → L3。不理解代码就进入 L2，只是机械 grep。

---

## 2. L0 代码审计

### 2.1 建立目标集（自行发现，不从本文读取）

审计开始时，AI 自行扫描项目目录结构，归纳出：

- **核心脚本**：包含主要业务逻辑的脚本（通常在 `scripts/` 下）
- **入口点**：CLI 命令、HTTP API 端点、定时任务入口
- **数据流**：数据从外部进入系统、在模块间传递、最终输出的路径
- **前端界面**：HTML 模板/页面文件

### 2.2 数据流追踪

选择至少 **2 条完整数据流**，从入口逐函数追踪到出口。

**如何选择**：
1. 一条**用户触发的主链路**（如"用户点击运行 → 后端计算 → 结果返回"）
2. 一条**自动化链路**（如"定时任务 → 数据抓取 → 写入存储"）

**追踪时关注**：
- 数据在函数间传递时，类型/单位/缩放是否一致？
- 有没有函数吞掉异常后返回空值，而上游未检查？
- 有没有"写入但无人读取"的中间状态？
- 错误路径是否和正常路径一样被认真对待？

### 2.3 逻辑审查

对追踪路径上的每个关键函数：

| 检查点 | 方法 |
|--------|------|
| **边界条件** | 输入为空、为负、为极端值时，函数行为是否正确？ |
| **fencepost** | 循环边界、日期范围、索引访问是否 off-by-one？ |
| **状态管理** | 全局变量/缓存是否在异常路径下也被正确清理？ |
| **并发安全** | 多个请求同时触发时，是否存在竞态？ |

### 2.4 冗余检测

- 是否存在两个函数做同一件事？
- 是否有被标记为废弃但代码体完整的逻辑？
- 是否有从未被调用的函数或常量？
- 同一阈值/配置值是否在多处硬编码？

发现冗余时，标注位置和建议，由用户决定处理方式。

### 2.5 前端审查

对每个 HTML 页面/模板：

| 维度 | 检查点 |
|------|--------|
| **交互反馈** | 每次用户操作是否有即时视觉反馈？ |
| **三态覆盖** | loading / empty / error 三种状态是否都有 UI 处理？ |
| **信息架构** | 重要信息首屏可见？相关控件分组清晰？ |
| **错误处理** | API 失败时用户看到友好提示还是白屏？ |
| **一致性** | 颜色/间距/字体是否统一？与项目整体风格是否匹配？ |

## 3. L1 边界审计

快速扫描。不应超过总审计时间的 15%。

- 凭证/密钥是否被硬编码在代码或文档中？
- **技能闭环性**（外部引用泄露）：
  - 是否有本地绝对路径？（`C:/Users/...`、`D:/...`、`file:///...`）
  - 是否引用了技能目录外部的文件？（`~/...`、`/home/...`、其他工作区路径）
  - 是否引用了外部工具/平台的具体名称，使文档无法在不同环境复用？
  - 所有引用必须能在技能目录内闭环解析；外部依赖只能是 `requirements.txt` 中的包或 `research/` 中的调研文档
- 是否有临时文件/备份文件遗留（`_working/`、`.bak`、`.old` 等）？
- 敏感配置文件是否被版本控制跟踪？

---

## 4. L2 一致性审计

**方法**：以代码为事实源，文档为被审核方。

1. 从 L0 中已建立的目标集中提取关键符号（函数名、端点路径、配置键名、因子名）
2. 在文档中 grep 这些符号
3. 不一致 → 记录，代码为准

**跨文档冲突**：若两份文档对同一事实描述不一致，以"最接近代码的文档"为准（运维手册 > 设计文档 > 概述文档）。无法判断时标注为"需人工核查"。

**退役清理**：从项目看板/配置中读取退役对象清单，在全部文档中 grep——退役对象不应以正文形式存在。允许一处集中记录"已退役"，不允许保留独立章节。

## 5. L3 质量审计

### 5.1 代码质量（10 分）

| 检查点 | 方法 |
|--------|------|
| 函数长度 | 标注 >200 行的函数（不强制拆分，只标注） |
| 命名一致性 | 同一概念在不同文件中是否用同一名称？ |
| 注释诚实 | 注释是否与代码实际行为一致？是否有误导性注释？ |

### 5.2 参数与可解释性（10 分）

| 检查点 | 方法 |
|--------|------|
| **决策可追溯** | 从策略输出能否反推"为什么选 A 不选 B"？关键因子的各自贡献是否可分解？ |
| **参数有据** | 关键参数的取值是否有研究/回测记录支撑？参数变更是否留下了变更理由？ |
| **敏感度** | 关键参数 ±20%，输出是否剧烈波动？若某参数极度敏感，是否在文档中标注？ |

### 5.3 文档质量（5 分）

仅检查核心文档（由 AI 自行判断哪些是核心文档）的关键问题：
- 符号与代码一致
- 退役内容清理
- 关键规则在首尾位置（非中段深埋）
- 指令格式可执行（"当 X → 做 Y"，而非"建议考虑 X"）

## 6. 评分标准

| 层 | 满分 | 扣分指南 |
|----|:---:|---------|
| L0 代码审计 | 40 | 数据流 bug（-10/个）、冗余逻辑（-5/处）、异常处理缺失（-5/路径）、前端 UX 缺陷（-3/个） |
| L1 边界 | 15 | 凭证泄露（-15 直接挂）、死文件（-3/个） |
| L2 一致性 | 20 | 符号不一致（-5/处）、退役对象残留（-5/处） |
| L3 质量 | 25 | 参数无解释（-5/项）、代码不可维护（-3/处）、文档不可操作（-3/处） |
| **总分** | **100** | |

**评级**：S(90+) 生产级 | A(75-89) 可用 | B(60-74) 需重大修改 | C(<60) 需重新设计

**证据要求**：
- 所有扣分必须附文件路径 + 行号。无位置引用的扣分无效。
- 修复建议使用"当前 → 改为"对照格式。
- 审计开始前，AI 预设"至少存在 3 个可改进点"（反确认偏误）。

## 7. 执行流程

### Step 0: 建立基线

**0.1 理解系统**：
1. 阅读项目入口文档 → 了解系统是什么、做什么
2. 阅读架构/设计文档 → 理解模块划分和数据流
3. 尝试启动系统 → 确认可运行，观察启动日志
4. 通读至少 2 个核心模块的完整代码

**0.2 建立对象清单**（自行从项目中提取，不从本文读取）：
- 活跃的功能模块/因子/Preset
- 已退役的功能模块/因子/Preset（从看板或配置中读取）
- 事实源映射：每类信息的权威来源是哪份文件

**完成标志**：输出对象清单后才进入 Step 1。

### Step 1: L0 代码审计

按 §2 方法执行。占总时间 50%+。

### Step 2: L1 边界审计

按 §3 清单逐项扫描。

### Step 3: L2 一致性审计

按 §4 方法交叉验证。

**反确认偏误检查点**（L2 开始前输出）：
```
基于 Step 0 基线，预计至少会发现：
  1. [退役对象残留]
  2. [符号名过期]
  3. [跨文档不一致]
若最终审计结论与此完全不同，在报告中解释原因。
```

### Step 4: 输出审计报告

```
══════════════════════════
审计报告 — <日期>
══════════════════════════

  概述: <一句话项目健康度>

  [L0] 代码审计: bug X / 冗余 X / 边界 X / 前端 X
  [L1] 边界审计: <N> 项
  [L2] 一致性审计: <N> 项
  [L3] 质量审计: XX/100 [评级]

  Top5 问题:
    1. [L0-bug] <文件:行号> — 描述 → 修复建议
    ...

  优先级: P0(阻塞发布) / P1(本版本) / P2(下版本)
══════════════════════════
```

## 8. LLM 认知定律

| 定律 | 内容 | 审计应用 |
|------|------|---------|
| **注意力衰减** | AI 对输入前 20% 和后 10% 记忆最强 | 最严重的问题放在报告第一条和最后一条 |
| **Token 预算** | 每千 token 占用约 3-5% 有效思考 | 不复制大段源码，用 `文件:行号` 引用 |
| **格式锚定** | "当前→改为"格式遵从度远高于"建议优化" | 修复方案统一用对照格式 |
