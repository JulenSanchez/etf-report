# 通用状态栏协议宿主

## 目标

本规则负责提供**可迁移的状态栏协议宿主**，用于在当前工作区内统一承接状态头显示、配置发现、配置选中与降级行为。

它不绑定任何具体业务主题，也不负责某个项目自己的需求看板、发布守卫或流程治理。

## 宿主职责

### 1. 固定显示协议

当当前对话命中某份状态栏配置时，AI 回复必须先输出状态抬头，且格式优先按以下 **4 段结构**组织：

```text
> {网络段} | {状态段} | {micro-stage} | {上下文}
────────────────────────────────────────────
正文从这里开始
```

其中：
- `网络段`：来自 `Display.network_marker`，例如 `🧠 Risk`、`📊 ETF`
- `状态段`：由**配置中已定义的 canonical state/action key 对应显示名**组成，例如 `🔍 审视`、`📌 需求`
- `micro-stage`：由 **自由生成的 `emoji + prompt`** 组成，用于表达当前片段的局部阶段、动作手感与节奏感，例如 `✅ 校验`、`🔄 回读`、`🧩 收口`
- `上下文`：当前主题下的右侧说明，例如 `时间状态机`、`骨架与肌理`
- 对于 `skill`、`module` 等具名主题配置，`Display.network_marker` 视为必填

补充说明：
- `状态段` 必须保持稳定，只能使用协议 / 配置中已定义的状态或动作，不得把 `micro-stage` 直接抬升替换为状态段
- `micro-stage` 可以更细、更活，允许 AI 按当前片段自由发挥 emoji 与短提示词，但应服务于可读性，不应喧宾夺主
- 只有 `workspace-default` 配置或协议内置配置，才允许省略 `网络段`，占用无命名空间的公共格式
- 推荐显示为：`> 🧠 Risk | 🔍 审视 | ✅ 校验 | 时间状态机`

要求：
- 第一行优先写成 `> {网络段} | {状态段} | {micro-stage} | {上下文}`
- 只有 `workspace-default` 配置与协议内置配置，才允许降级为 `> {状态段} | {micro-stage} | {上下文}`
- 第二行必须单独成行，且内容只能是完整分隔线 `────────────────────────────────────────────`
- 正文必须从第三行开始，不能和分隔线写在同一行
- 禁止把状态行、分隔线、正文压成同一行或同一段落
- 除 `workspace-default` 配置与协议内置配置外，缺少 `Display.network_marker` 的配置应视为不完整配置，不应占用公共格式
- `需求` 场景优先写成 `🧠 Risk | 📌 需求 | 🗂️ 梳理 | REQ-XXX | 标题/阶段` 这类四段式扩展
- `Bug` 场景优先写成 `🧠 Risk | 🐞 缺陷 | 🩺 定位 | BUG-XXX | 状态 | 严重度` 这类四段式扩展





### 2. 渲染安全约束

状态抬头必须优先使用 markdown 渲染安全写法，避免被渲染器折叠成单段文本：

```text
> {网络段} | {状态段} | {micro-stage} | {上下文}  
────────────────────────────────────────────  
正文从这里开始
```



补充约束：
- 第一行和第二行的行尾必须保留 markdown 硬换行（两个空格）
- 如果当前通道会自动裁掉行尾空格，则降级改用 `<br>` 明确断行
- 优先级：先保证用户看到三行，再追求源码形式最像三行

### 3. 配置发现与选中

在工作区内讨论某个主题时，宿主规则应按以下原则寻找活动配置：

1. 在当前工作区 `.codebuddy/` 范围内寻找状态栏配置候选
2. 候选文件优先包括：
   - 文件名为 `statusbar.config.md` 的文件
   - 或 frontmatter 中声明 `config_type: statusbar` 的 markdown 文件
3. 结合当前对话关键词、正在编辑/提及的路径、主题语义，选中最匹配的一份配置
4. 若多个配置同时命中，则按“更具体主题 > 显式路径命中 > 更高 priority > 工作区默认配置 > 协议内置兜底”的顺序决策

### 4. 活动配置持有与重判

选中活动配置后：
- 后续回复默认沿用该配置的 `States` / `Actions` / `Routing` / `Stacking` / `Display`
- 不要在每轮回复都重新全量搜索
- 仅在以下情况重判：
  - 用户明确切换到另一个主题 / 技能 / 模块
  - 当前配置明显无法解释对话内容
  - 配置文件被用户更新
  - 用户要求重新判断当前状态网络

### 5. 边界约束

本规则**只负责通用协议宿主能力**，不负责：
- 某个具体项目的需求看板规则
- 发布流程守卫
- 版本号维护
- 项目专属的入口 / 出口守卫

这些都应由项目自己的规则文件承担。

## 维护约定

- 调整固定三行状态头格式或渲染安全约束：改本文件
- 调整某个主题的状态 / 动作 / 路由：改对应主题的 `statusbar.config.md`
- 调整字段定义、发现协议、选中协议：改对应协议文档
